# bustaams 소비자 버스 요청 등록 화면 및 데이터 연동 통합 명세서

메인 홈페이지에 연동될 **소비자 버스 요청(스마트 입찰) 등록** 기능의 최종 설계 및 데이터 연동 명세입니다. 본 문서는 UI, 프론트엔드, 백엔드, DB 및 외부 API 연동을 아우르는 구현 가이드라인을 제공합니다.

---

## 1. 프론트엔드 (UI/UX) 설계 및 로직

### 1.1 메뉴 진입 및 인증 정책
- **위치:** 헤더(Header) → [이용고객전용] → [소비자 버스 요청 등록]
- **인증 가드 (Authentication Guard):** 
  - 비로그인 사용자가 메뉴 클릭 시 "로그인이 필요한 서비스입니다" 안내 후 로그인 모달(`LoginModal`)을 리다이렉트합니다.
  - 로그인 성공 후 인가된 **JWT Access Token**을 모든 API 요청 헤더(`Authorization: Bearer <Token>`)에 포함하여 전송합니다.

### 1.2 운행 일시 기반 자동 계산 로직 (UX)
사용자가 `START_DT`와 `END_DT`를 선택하면 프론트엔드 상태(State)에서 즉시 다음 값을 계산하여 UI에 반영합니다.
- **운행 일수 (Days):** `Math.ceil((END_DT - START_DT) / (1000 * 60 * 60 * 24))`
  - 예: 당일(1일), 1박 2일(2일)
- **숙박 박수 (Nights):** `Math.floor((END_DT - START_DT) / (1000 * 60 * 60 * 24))`
  - 예: 당일(0박), 1박 2일(1박)
- **자동 비용 업데이트:** 일수/박수가 계산되면 `식대(단가 * 3끼 * Days)`와 `숙박비(단가 * Nights)`가 인풋 필드에 즉시 기본값으로 세팅됩니다.
- **차량 대수 관련 3개 항목 관리:**
  1. **총 탑승인원 (PASSENGER_CNT):** 소비자 직접 입력
  2. **계산된 필요 차량 대수 (CALC_BUS_CNT):** `Math.ceil(PASSENGER_CNT / BUS_CAPACITY)` 자동 산출 및 표시 (읽기 전용 추천)
  3. **직접 입력한 필요 차량 대수 (BUS_CNT):** 소비자가 최종적으로 요청하는 대수 (기본값은 CALC_BUS_CNT와 동일하게 세팅하되 수동 수정 가능)

### 1.3 주소 검색 및 위경도(Lat/Lng) 처리
- **연동 API:** Kakao Postcode 및 Local API (Keyword 검색 지원)
- **로직:** 사용자가 장소명 또는 주소를 선택하면 해당 위치의 **위도(LAT)와 경도(LNG)**를 확보하여 상태값으로 유지합니다. (지도 미리보기에 활용 및 API 전송 필수)

## 2. 프론트엔드 ↔ 백엔드 API 파라미터 정의

### 2.1 [POST] /api/bid/estimate (경로 및 예상 비용 조회)
사용자가 등록 전에 대략적인 비용을 확인하기 위해 호출하는 API입니다.

| 파라미터 명 | 타입 | 필수 여부 | 설명 |
| --- | --- | --- | --- |
| `startLat`, `startLng` | Number | O | 출발지 위경도 좌표 |
| `endLat`, `endLng` | Number | O | 도착지 위경도 좌표 |
| `waypoints` | Array | X | 경유지 목록 (최대 3개, 각 항목에 `lat`, `lng` 포함) |
| `busType` | String | O | 버스 종류 (LARGE, PREMIUM 등) |

### 2.2 [POST] /api/bid/request (최종 요청 등록)
입찰 요청을 데이터베이스에 영구 저장하는 핵심 API입니다.

| 파라미터 명 (JSON Body) | 타입 | 제약 조건 | 설명 |
| --- | --- | --- | --- |
| `busType` | String | NOT NULL | 선택한 버스 모델 |
| `passengerCnt` | Number | Min: 1 | 탑승 인원 |
| `calcBusCnt` | Number | Min: 1 | 시스템 자동 계산 차량 대수 |
| `busCnt` | Number | Min: 1 | 소비자가 최종 확정한 필요 차량 대수 |

| `startDt` | DateTime | Future Only | 운행 시작 시간 (ISO8601) |
| `endDt` | DateTime | > `startDt` | 운행 종료 시간 |
| `roundTripYn` | String | 'Y' or 'N' | 왕복 여부 |
| `startAddr` / `startDetail` | String | NOT NULL | 장소명 및 상세 주소 |
| `startLat` / `startLng` | Number | Coordinate | 출발 좌표 |
| `endAddr` / `endDetail` | String | NOT NULL | 장소명 및 상세 주소 |
| `endLat` / `endLng` | Number | Coordinate | 도착 좌표 |
| `waypoints` | Array | Max Items: 3 | `[{addr, detail, lat, lng, order}]` |
| `mealPrice` / `lodgingPrice` | Number | >= 0 | 계산 완료된 비용 값 |
| `tipPrice` | Number | >= 0 | 기사님 팁 |
| `comment` | String | Max 1000자 | 세부 요구 사항 (Comment) |

## 3. DB 생성용 쿼리 (DDL - 성능 및 무결성 강화)

```sql
-- 1. 입찰 요청 메인 테이블
CREATE TABLE TB_BID_REQUEST (
    REQUEST_ID BIGINT AUTO_INCREMENT PRIMARY KEY,
    USER_ID VARCHAR(300) NOT NULL COMMENT '소비자 ID (TB_USER.USER_ID 참조 FK) - AES-256-GCM 암호화 저장, TB_USER와 동일 타입',
    BUS_TYPE VARCHAR(50) NOT NULL,
    BUS_FUEL_EFF DECIMAL(5, 2) DEFAULT 4.50,
    PASSENGER_CNT INT NOT NULL DEFAULT 1,
    CALC_BUS_CNT INT NOT NULL DEFAULT 1 COMMENT '계산된 필요 차량 대수',
    BUS_CNT INT NOT NULL DEFAULT 1 COMMENT '최종 확정 필요 차량 대수',

    START_DT DATETIME NOT NULL COMMENT '운행 시작 일시',
    END_DT DATETIME NOT NULL COMMENT '운행 종료 일시',
    ROUND_TRIP_YN CHAR(1) DEFAULT 'N' CHECK (ROUND_TRIP_YN IN ('Y', 'N')) COMMENT '왕복 여부',
    
    START_ADDR TEXT NOT NULL,
    START_DETAIL_ADDR TEXT,
    START_LAT DECIMAL(18, 10),
    START_LNG DECIMAL(18, 10),
    END_ADDR TEXT NOT NULL,
    END_DETAIL_ADDR TEXT,
    END_LAT DECIMAL(18, 10),
    END_LNG DECIMAL(18, 10),
    
    TOTAL_DISTANCE_KM DECIMAL(10, 2) DEFAULT 0.00,
    TOTAL_DURATION_MIN INT DEFAULT 0,
    REST_AREA_CNT INT DEFAULT 0,
    
    FUEL_PRICE_PER_L DECIMAL(10, 2) DEFAULT 1600.00,
    EST_FUEL_COST DECIMAL(15, 2) DEFAULT 0,
    TOTAL_TOLL_FEE DECIMAL(15, 2) DEFAULT 0,
    MEAL_PRICE DECIMAL(15, 2) DEFAULT 0,
    LODGING_PRICE DECIMAL(15, 2) DEFAULT 0,
    TIP_PRICE DECIMAL(15, 2) DEFAULT 0,
    INCIDENTAL_SUBTOTAL DECIMAL(15, 2) DEFAULT 0,
    COMMENT TEXT COMMENT '세부 요구사항',
    EST_TOTAL_SERVICE_PRICE DECIMAL(15, 2) DEFAULT 0 COMMENT '서버 자동 산출 예상 총 비용 (프론트 미전송, 백엔드 계산 후 INSERT)',
    
    STATUS VARCHAR(20) DEFAULT 'OPEN',
    REG_DT DATETIME DEFAULT CURRENT_TIMESTAMP,
    REG_ID VARCHAR(300),
    MOD_DT DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    MOD_ID VARCHAR(300),
    
    -- 인덱스 추가: 조회 속도 및 필터링 최적화
    INDEX IDX_BID_USER (USER_ID),
    INDEX IDX_BID_START_DATE (START_DT),
    INDEX IDX_BID_STATUS (STATUS)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='소비자 입찰 요청 메인';

-- 2. 경유지 상세 테이블
-- ⚠️ 경유지 최대 개수(3개) 제한은 DDL로 강제할 수 없으므로, 반드시 백엔드 Validation Layer(Joi/Zod)에서 검증해야 합니다.
CREATE TABLE TB_BID_WAYPOINT (
    WAYPOINT_ID BIGINT AUTO_INCREMENT PRIMARY KEY,
    REQUEST_ID BIGINT NOT NULL,
    WAYPOINT_ADDR TEXT NOT NULL,
    WAYPOINT_DETAIL_ADDR TEXT,
    LAT DECIMAL(18, 10),
    LNG DECIMAL(18, 10),
    SORT_ORDER INT NOT NULL,
    DIST_FROM_PREV DECIMAL(10, 2) DEFAULT 0,
    TOLL_FROM_PREV DECIMAL(15, 2) DEFAULT 0,
    DURATION_FROM_PREV INT DEFAULT 0,
    REG_DT DATETIME DEFAULT CURRENT_TIMESTAMP,
    REG_ID VARCHAR(300),
    INDEX IDX_WAYPOINT_REQ_ID (REQUEST_ID),
    CONSTRAINT FK_WAYPOINT_REQ FOREIGN KEY (REQUEST_ID) REFERENCES TB_BID_REQUEST(REQUEST_ID) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='입찰 요청 경유지 상세';
```

## 4. 백엔드 비즈니스 서버 로직

### 4.1 데이터 유효성 검사 (Validation Layer)
- **서버 검증 규칙:** 
  - `END_DT`는 반드시 `START_DT`보다 이후여야 합니다.
  - 좌표(`LAT`/`LNG`) 값은 한국 위도/경도 범위 내에 있어야 합니다 (위도: 33~39, 경도: 124~132).
  - `ROUND_TRIP_YN`은 반드시 `'Y'` 또는 `'N'` 값만 허용합니다.
  - 경유지(`waypoints`) 배열은 **최대 3개** 까지만 허용합니다.
  - `EST_TOTAL_SERVICE_PRICE`는 **백엔드 서버가 직접 산출**하는 값이며 프론트엔드에서 전송하지 않습니다. 서버 계산 결과가 0원 이하일 수 없습니다.

> 💡 **`EST_TOTAL_SERVICE_PRICE` 산출 주체:** 이 값은 프론트에서 보내는 파라미터가 아닙니다. 백엔드가 `EST_FUEL_COST + TOTAL_TOLL_FEE + MEAL_PRICE + LODGING_PRICE + TIP_PRICE`를 합산하여 `INCIDENTAL_SUBTOTAL` 및 `EST_TOTAL_SERVICE_PRICE`를 **INSERT 직전에 자동 계산**하여 DB에 저장합니다.

### 4.2 외부 API 연동 프로세스
1. **Opinet 연동:** 서버 부하 감소를 위해 실시간 유가 정보는 1시간 단위로 **캐싱(Caching)**하여 사용합니다.
2. **휴게소 POI 필터링:** 
   - Route 좌표 데이터를 기반으로 카카오 POI 검색 API를 호출합니다.
   - 검색 카테고리 '휴게소'에 해당하는 포인트만 필터링하여 유효한 개수를 산출합니다.

### 4.3 사후 처리 (Post-Process) 및 알림 정책
- **트랜잭션 Commit 후 즉시 실행:** 
  - 사용자에게 **알림톡(또는 SMS)** 발송: "입찰 요청이 정상 등록되었습니다."
  - 인근 지역 정회원 기사님들에게 **PUSH 알림** 발송: "새로운 입찰 요청이 도착했습니다!"

- **알림 발송 실패 정책 (Failure Policy):**
  - 알림 발송은 입찰 등록 트랜잭션과 **분리된 비동기(Async) 처리**로 실행합니다. 알림 발송 실패가 입찰 등록 자체를 롤백시켜서는 **절대 안 됩니다.**
  - 실패 시 **최대 3회 재시도(Retry)** 후에도 실패하면 실패 로그를 기록하고 관리자에게 알립니다.
  - 향후 메시지 큐(예: Cloud Tasks, Pub/Sub)를 도입하여 안정성을 강화할 수 있습니다.

### 4.4 암호화된 USER_ID 기반 조회(Join) 전략

> ⚠️ **중요:** `TB_BID_REQUEST.USER_ID`는 `TB_USER.USER_ID`와 동일하게 **AES-256-GCM**으로 암호화된 값을 저장합니다. 동일한 원문이라도 IV(초기화 벡터)가 매번 달라 암호문이 상이하므로, **일반 SQL JOIN이 불가**합니다.

- **현재 전략:** 입찰 요청 INSERT 시, 로그인 세션에서 이미 확보된 **암호화 상태의 `USER_ID` 원문(DB 저장값 그대로)**을 `TB_BID_REQUEST.USER_ID`에 동일하게 복사 저장합니다. 이렇게 하면 동일한 암호문이 양쪽 테이블에 존재하므로 **`WHERE` 및 `JOIN` 조건 매칭이 정상 작동**합니다.
- **향후 고도화:** 회원 수 및 입찰 건수가 방대해질 경우, 비암호화 대리키(`USER_SEQ BIGINT`) 또는 결정적 해시 인덱스 컬럼을 도입하여 조회 성능을 향상시키는 방안을 권장합니다.
