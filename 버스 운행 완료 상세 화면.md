# 버스 운행 완료 상세 화면 (`BusOperationCompletionDetails`)

**정본 DDL·컬럼 ID·타입**은 [`BusTaams 테이블.md`](./BusTaams%20테이블.md) — `TB_BUS_RESERVATION`, `TB_AUCTION_REQ`, `TB_AUCTION_REQ_BUS`, `TB_AUCTION_REQ_VIA`, `TB_USER`. 구 스키마의 `TRAVELER_UUID`·`DRIVER_UUID`·`RES_UUID`·`REQ_UUID`·`WRITER_UUID`(BINARY) 등은 **BusTaams 정본에서 제거**되었고, API·응답은 **`TRAVELER_ID`**, **`DRIVER_ID`**, **`RES_ID`**, **`REQ_ID`**, **`REQ_BUS_ID`** (각 `VARCHAR(10)` 등 정본과 동일) 기준으로 동작한다. 상세/목록 응답에서 **`resUuid`·`reqUuid`·`reqBusUuid`는 사용하지 않는다** (삭제).

부가: [`busTaams_web/BUSTAAMS_테이블 생성 쿼리 전체.md`](./busTaams_web/BUSTAAMS_테이블%20생성%20쿼리%20전체.md), `SERVER 환경.md` (로그인·`CUST_ID` 정책).

---

## 0. 식별자·필드 매핑 (DB ↔ REST/JSON)

| 구분 | DB 컬럼 | 타입(정본) | JSON / 쿼리 |
|------|---------|------------|-------------|
| 기사(예약) | `TB_BUS_RESERVATION.DRIVER_ID` | `varchar(10)` = `TB_USER.CUST_ID` | `driverId` / `userId` / `driverUuid` 중 1 (서버 `resolveCustIdByUserKey`로 `CUST_ID` 정합) |
| 완료 예약 | `RES_ID` | `varchar(10)` | `resId` (쿼리·응답) |
| 견적 마스터 | `REQ_ID` | `varchar(10)` | `reqId` (쿼리·응답) |
| 여행자(조인) | `r.TRAVELER_ID` / `res.TRAVELER_ID` | `varchar(10)` | (상세 응답: 복호화 `travelerName`만) |
| 견적 버스 | `REQ_BUS_ID` | `varchar(10)` | `auctionReqBuses[].reqBusId` |
| 상태 | `DATA_STAT` = `'DONE'` | | `dataStat` |
| 리뷰(참고) | `TB_TRIP_REVIEW.WRITER_ID` / `DRIVER_ID` | `varchar(10)` | 본 API 미포함 |

**채번(0-패딩):** `BusTaams 테이블.md` §채번 — `RES_ID`·`REQ_ID`·`REQ_BUS_ID`·`VIA_ID` 등 `VARCHAR(10)` 순번은 **왼쪽 0 채움** (`padStart(10, '0')` 등). **`TB_USER.USER_ID`**(로그인 ID)는 별도 정책.

## 3. 프론트엔드

| 파일 | 역할 |
|------|------|
| `BusOperationCompletionDetails.jsx` | 상세 모달, `GET /api/bus-operation-completion-details` |
| `vehicleLabels.js` | `getVehicleLabel` — `BUS_TYPE_CD` 표시 |
| `BusOperationCompletionList.jsx` | 목록에서 `driverId`·`resId`·`reqId` 전달 |

### 3-1. 레이아웃 (요약)

- **좌(넓은 화면 `lg:col-span-5`):** 「여행자 등록 정보」+「요청 차량」
- **우(`lg:col-span-7`):** 「왕복 운행 타임라인」(`timeline[]` — 점선·노드, `START_NODE`/`END_NODE` 강조)

### 3-2. 표시 데이터

| 화면 라벨 | 데이터 |
|-----------|--------|
| 여행자 성명 | `travelerName` (`TB_USER.USER_NM` 복호화, 없으면 `—`) |
| 탑승 인원 | `passengerCnt` — `PASSENGER_CNT` |
| 여정 제목 | `tripTitle` |
| 출발/도착 일시 | `startDt`, `endDt` |
| 여행자 요청 금액 | `travelerRequestKrw` — 마스터 `REQ_AMT` 또는 `TB_AUCTION_REQ_BUS` 합(`TOLLS_AMT`+`FUEL_COST`) |
| 버스기사 금액 | `driverBidKrw` — `DRIVER_BIDDING_PRICE` |
| 요청 차량 | `auctionReqBuses[]` — `reqBusId`, `busTypeCd`, `reqBusCnt`, `reqAmtKrw`, `sortOrder` |

**비표시:** 이 API는 `startAddr` / `endAddr`·지도·결제 상세·`TB_TRIP_REVIEW` 를 내려주지 않음.

### 3-3. Props (현행)

`open`, `onClose`, **`driverId`**, **`resId`**, **`reqId`**. (구 `driverUuid` 전용 prop·`resUuid` 쿼리 **미사용**)

요청 URL: `driverId=…`, `resId=…` 및/또는 `reqId=…` (값은 0패딩 ID일 수 있음; UUID가 아닐 수 있음 → 서버 쪽에서 소문자 정규화는 **식별에만** 사용).

## 4. REST API

| | |
|---|---|
| **GET** | `/api/bus-operation-completion-details` |
| **구현** | `busTaams_server/routes/busOperationCompletionDetails.js` |

### 4-1. Query (Request)

| 파라미터 | 필수 | 설명 |
|----------|------|------|
| `driverId` | 1개 필수* | `TB_USER.USER_ID` / `CUST_ID` / 예전 키 이름과 동일 의미 — `resolveCustIdByUserKey` → `DRIVER_ID` |
| `userId` | ↑ 동의어 | |
| `driverUuid` | ↑ 동의어 | (이름만 legacy; 값은 UUID가 아닐 수 있음) |
| `resId` | `resId` 또는 `reqId` | `TB_BUS_RESERVATION.RES_ID` (쿼리 시 소문자 `trim` 정규화) |
| `reqId` | 중 하나 | `TB_AUCTION_REQ.REQ_ID` — `resId` 없이 견적만으로 찾을 때 |

\* `driverId`·`userId`·`driverUuid` 중 **하나**.

**400:** 기사·식별 누락, `resId`·`reqId` 모두 없음.  
**404:** `DATA_STAT = 'DONE'` + 기사 일치 + 키 일치인 행 없음.  
**500:** DB 오류.

### 4-2. Response 200 (필드)

| 필드 | 타입 | 설명 |
|------|------|------|
| `travelerName` | string\|null | `USER_NM` 복호화 |
| `tripTitle` | string\|null | `TRIP_TITLE` |
| `startDt` | string\|null | `START_DT` |
| `endDt` | string\|null | `END_DT` |
| `passengerCnt` | number | `PASSENGER_CNT` |
| `travelerRequestKrw` | number | 여행자 측 요청 금액(원) |
| `driverBidKrw` | number | `DRIVER_BIDDING_PRICE` |
| `auctionReqBuses` | array | 아래 |
| `timeline` | array | `viaOrd`, `viaType`, `viaAddr`, `lineText`, `nodeKind` |
| `resId` | string\|null | `RES_ID` |
| `reqId` | string\|null | `REQ_ID` |
| `dataStat` | string | `DONE` |

`auctionReqBuses[]` 요소:

| 필드 | 설명 |
|------|------|
| `reqBusId` | `REQ_BUS_ID` (레거시 DB 폴백에서도 `reqBusId` 키로 통일) |
| `busTypeCd` | `BUS_TYPE_CD` |
| `reqBusCnt` | 대수 |
| `reqAmtKrw` | 행 금액(원) |
| `sortOrder` | 표시 순서(1..n) |

### 4-3. 스키마 폴백 (서버 내부)

1. **레거시 SQL** — `RES_STAT`·`RES_UUID`·`DRIVER_UUID`·`REQ_UUID` 등 **구 BINARY/UUID** 조인. (운영 `BusTaams`는 컬럼이 없을 수 있어) **`ER_BAD_FIELD_ERROR`(1054)** 이면 다음 단계로.
2. **ARCH(BusTaams):** `DATA_STAT = 'DONE'`, `DRIVER_ID` / `RES_ID` / `REQ_ID`, `REQ_ID` 기준 `TB_AUCTION_REQ_VIA`·`TB_AUCTION_REQ_BUS`. 기사 키는 `USER_UUID` 서브쿼리(세션이 UUID일 때)와 `DIRECT_ID = ?` **조합**을 순차 시도(`tryArchDetailQueries`).

`TB_AUCTION_REQ_VIA`: `REQ_ID`로 조회, 컬럼 없으면 `REQ_UUID`·`UUID_TO_BIN` 경로(마이그레이션 중).

## 5. BackEnd / `server.js`

- 운행 완료 **목록·상세**는 `INSERT/UPDATE/DELETE` 없이 **SELECT**만 수행.
- DML/필드명은 `BusTaams 테이블.md` **Field(컬럼명)** 및 위 표와 일치. 구식 BINARY/UUID 전용 쿼리는 **에러 시 폴백**(`busOperationCompletionDetails.js` 내 `SQL_*_LEGACY`).

**공용:** `lib/resolveCustIdByUserKey.js`, `server.js` 상 `require` — 기사 `USER_ID` → `CUST_ID` → `DRIVER_ID` 정합.
