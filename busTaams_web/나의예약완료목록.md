# 나의 예약 완료 목록 개발 가이드

성공적으로 확정된 프리미엄 버스 여정을 확인하고 관리하는 화면입니다. 입찰 진행 중인 건과 확정된 건을 통합 관리합니다.

## 1. 개요
- **목적**: 결제 및 확정이 완료된 여정 조회, 배정된 기사님과의 소통(채팅), 결제 처리 및 개별/전체 취소 관리.
- **진입 경로**: 고객 메인 대시보드 > [나의 예약 완료 목록] (또는 확정 내역)

## 2. 주요 기능 및 UI 구성
### A. 예약 목록 (Bidding & Confirmed List)
- **필수 조회 조건**: `TB_AUCTION_REQ` 테이블에서 `DATA_STAT IN ('AUCTION', 'CONFIRM')` 상태인 여정 조회.
- **상태별 UI 차별화**:
    - **견적 입찰 진행 중**: `DATA_STAT === 'AUCTION'`일 경우 주황색 점멸 인디케이터 표시.
    - **확정된 여정**: `DATA_STAT === 'CONFIRM'`일 경우 초록색 고정 인디케이터 표시.
- **데이터 조인 (Core Logic)**: 
    - `TB_AUCTION_REQ` (여정 마스터)
    - `TB_AUCTION_REQ_BUS` (요청 차량 정보)
    - `TB_BUS_RESERVATION` (입찰/확정 정보 - **LEFT JOIN**을 통해 견적 대기 상태도 노출)
- **주요 액션 버튼**:
    - **결제하기**: 확정된 차량(`CONFIRM`)에 대해 개별 결제 페이지 연동.
    - **채팅하기**: 확정된 기사님과 1:1 대화 연결 (아이콘 버튼).
    - **전체 예약취소**: 여정 전체를 취소하고 위약금 및 사유 입력 모달 호출.
    - **버스취소/변경**: 여정 내 특정 차량만 개별적으로 취소하거나 변경.

### B. 디자인 가이드라인 (Design Aesthetics)
- **영문 디자인 배제**: 모든 UI 텍스트는 한글로 표기 (예: ID → 예약번호, TOTAL QUOTE → 총 견적 금액).
- **프리미엄 룩앤필**: `RESERVATION DETAIL` 배너, `견적 상세 내역` (Slate-900 다크 카드) 등 고급스러운 UI 적용.
- **상태 피드백**: '취소됨' 상태의 차량은 60% 투명도 및 그레이스케일 처리로 시각적 구분.

## 3. 기술적 요구사항
- **ID 정합성 (Critical)**: `TB_AUCTION_REQ_BUS.REQ_BUS_SEQ`는 숫자형이나, `TB_BUS_RESERVATION.REQ_BUS_ID`는 10자리 문자열이므로 **`LPAD(SEQ, 10, '0')`**를 사용하여 조인 및 업데이트를 수행해야 함.
- **이력 관리**: 취소 시 `TB_USER_CANCEL_HIST`에 사유와 증빙 서류명을 기록하고, `TB_USER_CANCEL_MANAGE`에서 고객별 취소 카운트(전체/부분)를 갱신함.

## 4. 최근 수정 사항 (2026-04-28)
- **상태 구분 구현**: 대시보드 목록에서 '입찰 중'과 '확정' 상태를 시각적으로 분리.
- **데이터 무결성 수정**: `REQ_BUS_ID` 패딩 이슈(LPAD) 해결을 통해 견적 금액이 0원으로 표시되던 버그 수정.
- **UI 보완**: 결제하기 버튼 추가, 채팅 기능 이벤트 연동, 영문 라벨 전체 한글화 완료.
- **취소 로직 강화**: 증빙 파일 업로드 기능 및 부분 취소 통계 반영 로직 서버 연동.

---
> [!IMPORTANT]
> **데이터 정합성**: 한 여정에 여러 대의 버스가 있을 경우, 각 버스마다 확정된 기사가 다를 수 있습니다. 반드시 `REQ_ID`와 `REQ_BUS_SEQ`를 조합하여 정확한 매칭을 보장해야 합니다.
