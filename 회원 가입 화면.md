# bustaams 회원 가입 모달 (팝업) 구조 및 데이터 연동 사전 계획

메인 홈페이지에 연동될 **회원 가입 팝업 모달**의 설계 및 프론트-백엔드-DB 간의 데이터 정합성 검토 내역입니다. 본 문서는 코드를 작성하기 전 아키텍처 연동 가능성을 점검하는 용도입니다.

---

## 1. 프론트엔드 (UI) 메뉴 연결 계획
- **위치:** `busTaams_web` 프론트엔드 환경
- **작동 방식:** Bustaams 홈페이지 헤더(Header) 영역의 "회원 가입" 버튼 클릭 시, 전체 화면 페이지 이동이 아닌 현재 화면 위에 띄워지는 **모달(Modal) 팝업** 형태로 회원 가입 화면 창이 오버레이 노출됩니다.
- **디자인 정책:** 사전에 정의된 **[ARCHITECTURE.md](./ARCHITECTURE.md)** (The Radiant Traveler)의 No-Line Rule 및 브랜드 컬러(청록/주황)가 완벽하게 적용되도록 HTML 태그들을 React 컴포넌트로 리팩토링할 예정입니다.

---

## 2. 프론트엔드 UI ↔ DB 스키마(TB_USER) 간 데이터 매핑

제시해 주신 `TB_USER` 테이블 스키마와 프론트엔드 HTML 구조 간의 데이터 연동은 **완벽하게 문제없이 1:1 대응하여 호환(연동)** 가능합니다.

| 프론트엔드 (UI 입력 필드) | 백엔드 (Node.js 전송 데이터) | DB 컬럼 (TB_USER) | 설명 / 비고 |
| --- | --- | --- | --- |
| **Email** (이메일 텍스트박스) | `userId`, `email` | `USER_ID`, `EMAIL_ADDR` | 일반 가입 시 아이디 겸 이메일로 사용 (PK) |
| **Password** (비밀번호) | `password` | `PASSWORD` | 사용자 입력 비밀번호 (서버에서 단방향 암호화 후 Insert) |
| **Name** (이름) | `userName` | `USER_NM` | 사용자 실명 (NOT NULL) |
| **Phone** (휴대폰 번호) | `phoneNo` | `PHONE_NO` | 휴대폰 번호 (NOT NULL) |
| **Verification Code** (인증번호) | (서버 인증 로직 통과 여부) | `AUTH_VERIFY_YN` | 문자 인증이 완료되었을 경우 서버가 'Y'로 저장 |
| **Kakao / Naver Login** | `snsType`, `snsId` | `SNS_TYPE`, `SNS_ID` | 소셜 로그인 응답에서 넘겨받은 벤더사 식별자와 고유 ID |
| **Terms (마케팅 동의)** | `marketingAgreeYn` | `MKT_AGREE_YN` | 선택 약관(마케팅) 체크 박스 여부 ('Y' 또는 'N') |

*시스템 자동 기입 항목:* `USER_TYPE`(사용자 유형), `USER_STAT_CD`(가입/탈퇴 상태), `REG_DT`/`MOD_DT`(생성 시간).

---

## 3. 백엔드(Node.js) 설계 보완 체크리스트

백엔드 서버(`busTaams_server`) 구축 시 다음 중요 비즈니스 로직을 API 및 DB 제약 조건으로 구현해야 프론트엔드와 정상 동작합니다.

### 1) 소셜 로그인 대응 통신 (`SNS_TYPE`, `SNS_ID`)
- 일반 이메일 가입 폼을 숨기고 즉시 소셜 연동 API를 호출하는 React 로직 구현 필요.
- 프론트에서 넘어온 `SNS_ID`를 Node.js가 수신한 뒤, 테이블 조회하여 기존 가입자인지 신규인지 구별하여 `TB_USER`에 INSERT 작업 수행.

### 2) 비밀번호 예외(NULL) 처리
- 일반 가입자는 반드시 프론트엔드에서 비밀번호 정규식(문자+숫자 특수문자 8자리 이상 등)을 체크한 뒤 전송하고 백엔드가 저장.
- 반면 카카오/네이버 가입자는 데이터베이스 `PASSWORD` 컬럼에 `NULL` 값이 들어갈 수 있어야 함. (또는 임의의 난수로 채우는 백엔드 전처리 로직 필요)

### 3) 독립적인 점유 인증 (`AUTH_VERIFY_YN`)
- 회원 가입 최종 완료 버튼(`Create Account`)을 누르기 *이전에*, 휴대폰 인증번호 API를 먼저 호출하여 인증이 완료되었는지 확인하는 프론트엔드 방어 로직이 필요.
- 인증 여부가 검증된 세션 혹은 토큰에서만 백엔드가 `AUTH_VERIFY_YN = 'Y'`로 저장 조치.

### 4) 회원 탈퇴 및 상태 파악 (`USER_STAT_CD`)
- React 화면에서 회원이 탈퇴를 요청했을 때, 실제 DB Row를 `DELETE`하는 것이 아니라 백엔드가 `UPDATE TB_USER SET USER_STAT_CD='90'`으로 상태 값을 변경하는 설계 채택 필요 (법적 보관 기준 충족).
