# bustaams 회원 가입 모달 (팝업) 구조 및 데이터 연동 사전 계획

메인 홈페이지에 연동될 **회원 가입 팝업 모달**의 설계 및 프론트-백엔드-DB 간의 데이터 정합성 검토 내역입니다. 본 문서는 코드를 작성하기 전 아키텍처 연동 가능성을 점검하는 용도입니다.

---

## 1. 프론트엔드 (UI) 메뉴 연결 계획 및 반영 내역
- **위치:** `busTaams_web` 프론트엔드 환경 (`App.jsx` 내 `SignUpModal` 컴포넌트로 완벽 구현 완료)
- **작동 방식:** Bustaams 홈페이지 헤더(Header) 측면의 "회원가입" 전용 버튼 클릭 시, 뒷배경이 블러(`backdrop-blur`) 처리되며 화면 중앙에 모달 팝업 창이 부드럽게 노출됩니다.
- **디자인 정책 반영 내역:** 
  - 사전에 정의된 **[ARCHITECTURE.md](./ARCHITECTURE.md)** (The Radiant Traveler)의 No-Line Rule 및 커스텀 그림자(`shadow-ambient`) 규칙을 엄격하게 적용하여 리팩토링했습니다.
  - **소셜 로그인 버튼 디자인 특화:** 사용자 인지가 쉽도록 '카카오 회원 간편 가입' 버튼은 고유의 노란색 바탕(`bg-[#FEE500]`)과 검정 글씨로, '네이버 회원 간편 가입' 버튼은 고유의 녹색 바탕(`bg-[#03C75A]`)에 흰 글씨로 적용해 디자인 시인성을 대폭 끌어올렸습니다.

## 2. 프론트엔드 UI ↔ DB 스키마(TB_USER) 간 데이터 매핑

제시해 주신 `TB_USER` 테이블 스키마와 프론트엔드 HTML 구조 간의 데이터 연동은 **완벽하게 문제없이 1:1 대응하여 호환(연동)** 가능합니다.

| 프론트엔드 (UI 입력 필드) | 백엔드 (Node.js 전송 데이터) | DB 컬럼 (TB_USER) | 설명 / 비고 |
| --- | --- | --- | --- |
| **-** | - | `USER_UUID` | 시스템 관리용 고유 식별자 (BINARY 16) |
| **Email** (이메일 텍스트박스) | `userId`, `email` | `USER_ID_ENC` | 일반 가입 시 아이디 겸 이메일로 사용 (AES-256 암호화) |
| **Password** (비밀번호) | `password` | `PASSWORD` | bcrypt 단방향 암호화 |
| **Confirm Password** (비밀번호 확인) | - | - | 프론트엔드 UI 단에서 실시간 비교 검증 |
| **Name** (이름) | `userName` | `USER_NM` | 사용자 실명 (AES-256 암호화) |
| **Phone** (휴대폰 번호) | `phoneNo` | `HP_NO` | 휴대폰 번호 (AES-256 암호화) |
| **Verification Code** (인증번호) | (서버 인증 로직 통과 여부) | `SMS_AUTH_YN` | 문자 인증 완료 시 'Y' 저장 |
| **Kakao / Naver Login** | `snsType`, `snsId` | `SNS_TYPE`, `USER_ID_ENC` | 소셜 로그인 식별자와 고유 ID |
| **Terms (마케팅 동의)** | `marketingAgreeYn` | - | (`TB_USER_TERMS_HIST` 이력 테이블로 매핑됨) |
| **Signature (전자 서명)** | `signature` (Base64) | `SIGN_FILE_UUID` | (`TB_USER_TERMS_HIST` 이력 테이블로 매핑됨) |

*시스템 자동 기입 항목:* `USER_UUID`, `USER_TYPE`(사용자 유형), `USER_STAT`(가입/탈퇴 상태), `JOIN_DT`(가입 일시).

### [참고] 최종 병합된 TB_USER 생성 DDL (Create Query)
```sql
CREATE TABLE TB_USER (
    USER_UUID BINARY(16) NOT NULL PRIMARY KEY COMMENT '시스템 관리용 고유 식별자 (UUID)',
    USER_ID_ENC VARCHAR(255) NOT NULL COMMENT '사용자 입력 ID / SNS ID (AES-256 양방향 암호화)',
    SNS_TYPE ENUM('NONE', 'KAKAO', 'NAVER') DEFAULT 'NONE' COMMENT '간편로그인 타입',
    USER_TYPE ENUM('TRAVELER', 'DRIVER', 'PARTNER') NOT NULL COMMENT '회원구분',
    PASSWORD VARCHAR(255) NOT NULL COMMENT '비밀번호 (단방향 암호화)',
    USER_NM VARCHAR(255) NOT NULL COMMENT '회원성명 (AES-256 양방향 암호화)',
    HP_NO VARCHAR(255) NOT NULL COMMENT '전화번호 (AES-256 양방향 암호화)',
    EMAIL_ENC VARCHAR(255) COMMENT '이메일 주소 (AES-256 양방향 암호화)',
    SMS_AUTH_YN ENUM('Y', 'N') DEFAULT 'N' COMMENT 'SMS 문자 인증 여부',
    RECOM_CODE BINARY(16) COMMENT '추천인 코드 (영업파트너의 USER_UUID)',
    JOIN_DT DATETIME DEFAULT CURRENT_TIMESTAMP COMMENT '가입 일시',
    USER_STAT ENUM('ACTIVE', 'LEAVE', 'BANNED') DEFAULT 'ACTIVE' COMMENT '상태',
    UNIQUE KEY UK_USER_ID (USER_ID_ENC)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT ='통합 회원 마스터 테이블';
```

### [필독] 약관 동의 이력 추적 및 버전 관리 아키텍처 (Terms History)
회원이 특정 시점에 어떤 내용(버전)의 약관에 동의했는지를 기록으로 남기지 않으면 차후 법적 효력이나 분쟁 발생 시 치명적인 리스크가 됩니다. 따라서 `TB_USER`의 `MKT_AGREE_YN` 단일 컬럼만으로는 충분치 않으며, 다음 두 개의 약관 전용 테이블을 신설하여 이력을 완벽히 추적해야 합니다.

#### 1) 약관 마스터 테이블 (`TB_TERMS_MASTER`)
서비스에 존재하는 모든 약관의 내용과 '버전(Version)', '시행일자'를 중앙 관리하는 테이블입니다.

```sql
CREATE TABLE TB_TERMS_MASTER (
    TERMS_ID INT AUTO_INCREMENT PRIMARY KEY COMMENT '약관 고유 식별자',
    TERMS_TYPE VARCHAR(20) NOT NULL COMMENT '약관 종류 (SVC, PRIVACY, LOCATION, MKT, DRIVER)',
    TERMS_VERSION VARCHAR(10) NOT NULL COMMENT '약관 버전 (예: v1.0, v2.1)',
    REQUIRE_YN CHAR(1) DEFAULT 'Y' COMMENT '필수 동의 여부 (Y/N)',
    CONTENT_URL VARCHAR(500) COMMENT 'GCS에 저장된 약관 전문(PDF/MD) 물리 파일 접근 경로 URL',
    ENFORCE_DT DATE NOT NULL COMMENT '해당 버전 약관의 실제 시행 일자',
    REG_DT DATETIME DEFAULT CURRENT_TIMESTAMP COMMENT '약관 등록 일시'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='서비스 공통 약관 마스터 테이블';
```

#### 2) 회원 약관 동의 기록 테이블 (`TB_USER_TERMS_HIST`)
회원 가입이나 약관 개정 후 재동의를 받을 때마다 회원이 어떤 버전에 언제 동의했는지를 '영구 누적(Insert-Only)' 하는 핵심 법적 이력 테이블입니다.

```sql
CREATE TABLE TB_USER_TERMS_HIST (
    HIST_ID BIGINT AUTO_INCREMENT PRIMARY KEY COMMENT '이력 고유 번호',
    USER_UUID BINARY(16) NOT NULL COMMENT '동의한 회원 식별자 (TB_USER 참조 FK)',  
    TERMS_ID INT NOT NULL COMMENT '동의했던 특정 버전의 약관 ID (TB_TERMS_MASTER 참조 FK)',
    SIGN_FILE_UUID BINARY(16) COMMENT '동의의 법적 증거인 전자 서명 파일 식별자 (TB_FILE_MASTER 참조 FK)',
    AGREE_YN CHAR(1) NOT NULL COMMENT '동의 여부 (Y/N)',
    AGREE_DT DATETIME DEFAULT CURRENT_TIMESTAMP COMMENT '사용자 실제 약관 동의/거절 행위 일시',
    CLIENT_IP VARCHAR(45) COMMENT '동의 당시 접속 IP 주소 (Auditing 용도)'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='회원 약관 동의 상세 이력 테이블';
```

**[프론트/백엔드 연동 운영 가이드]**
1. 프론트엔드는 회원가입 창이 마운트될 때 `TB_TERMS_MASTER`에서 시행 중인 최신 버전의 `TERMS_ID` 목록을 API로 조회해 약관 체크박스를 그립니다.
2. 회원이 [가입 완료하기] 버튼을 누르면, 회원 정보와 함께 "약관 매핑 배열" + "서명 캔버스 이미지(Base64)"가 백엔드 서버로 전송됩니다.
3. 백엔드는 스토리지에 PNG 파일을 먼저 업로드하여 고유한 원본 파일 식별자 `SIGN_FILE_UUID`를 획득합니다.
4. 백엔드는 `TB_USER` Insert 완료 직후, `TB_USER_TERMS_HIST` 테이블에 회원이 제출했던 각 약관(`TERMS_ID`) 개수만큼 레코드를 Insert 하며, 이 때 모든 레코드에 방금 획득한 `SIGN_FILE_UUID`를 할당하여 "특정 내용에 대한 서명 동의 기록"을 완벽하게 Binding 결속시킵니다.

## 2.5 [중요] 카카오/네이버 소셜 로그인 단계적 도입(Phase 1, 2) 전략

현재 설계는 **"당장은 일반(이메일/비밀번호) 가입으로 먼저 오픈하고, 추후 카카오/네이버 연동을 매끄럽게 추가할 수 있도록"** 대비된 확장성 있는 플러그 앤 플레이(Plug-and-Play) 구조입니다. 소셜 로그인을 당장 배제하더라도 시스템을 전혀 뜯어고칠 필요가 없는 이유는 다음과 같습니다.

### 1) DB 스키마의 완벽한 미래 호환성 (TB_USER)
- **비워둬도 무방한 공간:** 이미 `TB_USER` 테이블에 `SNS_TYPE`과 `SNS_ID` 컬럼이 마련되어 있습니다. 당장은 이 자리를 비워두고(기본값 'NONE') 사용하다가, 차후 소셜 가입 기능이 추가되면 해당 컬럼에 데이터만 채워 넣으면 끝입니다.
- **예외 처리된 PASSWORD:** `PASSWORD` 컬럼 역시 향후 소셜 로그인 유저(비밀번호가 없는 유저)가 들어올 것을 완벽히 대비해, 빈 값(`NULL`)도 허용되도록 예외 처리를 갖춘 상태로 설계되었습니다. 일반 가입 유저는 기존처럼 암호화된 비밀번호를 저장하면 됩니다.

### 2) 백엔드 문서의 미래 가이드라인 활용
- 본 문서 하단의 '3. 백엔드 설계 보완 체크리스트'에 기재된 [OAuth 2.0 기반 토큰 관리 및 2-Step 가입 프로세스]는 나중에 소셜 로그인을 붙이게 될 때 백엔드/프론트엔드 파트너가 혼동하지 않도록 미리 작성해 둔 **훌륭한 미래 설계도(Blueprint)** 역할을 합니다.
- 당장 구현하지 않더라도 설계도는 버리지 않고 유지하여, 향후 업데이트 시 곧바로 로직을 추가 활용하면 됩니다.

### 3) 프론트엔드 UI의 독립적 확장성
- 프론트엔드의 화면 단에서도 소셜 연동 버튼은 주요 가입 흐름에 종속되지 않습니다.
- 지금 당장은 모달 UI에서 노란색(카카오), 녹색(네이버) 버튼을 잠시 숨김(주석 처리) 해두거나 '준비 중' 모달을 띄우기만 하면 됩니다. 
- 추후 백엔드 소셜 연동 API가 완성되면, 즉시 UI 버튼 이벤트만 연결해주어(Switch-On) 기존 가입 폼 구조를 전혀 갈아엎지 않고도 유연하게 기능 확장이 가능합니다.

## 3. 백엔드(Node.js) 설계 보완 체크리스트

백엔드 서버(`busTaams_server`) 구축 시 다음 중요 비즈니스 로직을 API 및 DB 제약 조건으로 구현해야 프론트엔드와 정상 동작합니다.

### 0) 외부 소셜 디벨로퍼스 앱 등록 (사전 준비 필수)
자사 서버가 카카오/네이버 로그인 API와 통신하려면, 벤더사(카카오/네이버)로부터 "내가 합법적인 busTaams 서버다"라고 인증받는 자격 증명서(Key)를 먼저 발급받아야 합니다.
1. **Kakao Developers / Naver Developers 접속:** 회사(busTaams) 혹은 대표 계정으로 각 포털의 개발자 센터에 접속하여 새로운 애플리케이션(App) 정보를 등록합니다.
2. **필수 키(Key) 획득:** 등록이 완료되면 고유한 `Client ID` (REST API 키)와 `Client Secret` (보안 비밀키)이 발급됩니다. 이 키가 있어야 OAuth Access Token을 받아올 수 있습니다.
3. **Redirect URI 설정:** 유저가 소셜 로그인을 마치고 토큰과 함께 돌아올 자사 백엔드 도착지 주소(예: `http://localhost:8080/api/auth/kakao/callback`)를 벤더사 개발자 페이지에 미리 등록해야 합니다. (미등록 시 해킹 시도로 간주되어 에러가 발생합니다.)
4. **환경 변수(.env) 보관:** 발급받은 키 값들은 대단히 민감한 탈취 대상이므로 절대 Github 소스 파일에 적어두지 말고, 백엔드 서버 내의 `.env` 파일에 `KAKAO_CLIENT_ID="..."` 형태로 텍스트 보관하여 보호해야 합니다.

### 1) 소셜 로그인 대응 및 보안 토큰(JWT) 파이프라인 설계
- **설계 충돌 방지:** 카카오/네이버 연동 성공 직후(이름, 이메일 수신 시) 백엔드에서 즉시 `TB_USER`에 `INSERT` 하면 **절대 안 됩니다**. (필수 값인 전화번호, 서명 파일 등이 확보되지 않아 법적 시스템 무결성이 붕괴됩니다.)

**[OAuth 2.0 기반 토큰 관리 및 2-Step 가입 프로세스]**
1. **OAuth 인증 (Back-channel):** 카카오/네이버 로그인 시 발급되는 Authorization Code를 백엔드가 전달받아 직접 각 벤더사(카카오 등) 서버와 통신하여 **OAuth Access Token**을 발급받고, 이를 통해 `SNS_ID`, `이름`, `이메일` 정보를 추출합니다. 
2. **분기 및 자사 토큰(App JWT) 발행:**
   - **기존 회원의 경우 (1초 간편 로그인):** `TB_USER`에 `SNS_ID`가 존재한다면 추가 가입 절차가 필요 없으므로, 백엔드는 스스로 서명한 **busTaams 전용 JWT(Access / Refresh Token)**를 발행(Issue)하여 프론트에 넘겨주며 즉시 로그인 처리를 완료합니다.
   - **신규 회원의 경우 (임시 세션 발급):** 미가입자라면 `TB_USER`에 강제로 넣지 않고, 이메일과 `SNS_ID` 정보를 암호화해 담은 짧은 수명의 **임시 가입 토큰(Temp Signup JWT, 유효기간 10분)**을 생성해 프론트로 넘겨줍니다. 
3. **화면 렌더링 (보안):** 프론트엔드는 이 임시 토큰을 바탕으로 회원가입 모달을 띄우고, 이메일과 이름을 자동 기입하여 수정하지 못하게 잠금(Lock) 처리합니다.
4. **최종 승인 (검증):** 사용자가 **휴대폰 인증, 약관 동의, 서명 패드** 과정을 완료하고 [가입 완료하기] 폼 제출 시, 이전에 발급받았던 **임시 가입 토큰(Temp Signup JWT)**을 백엔드로 다시 함께 쏴줍니다.
5. **토큰 소모 및 권한 획득:** 백엔드는 이 임시 토큰의 무결성을 검증한 뒤(위변조 및 시간 초과 방어), 비로소 정상적인 절차임을 확정 짓고 `TB_USER`와 `TB_USER_TERMS_HIST`에 완전한 회원가입 데이터를 Insert 합니다. 그 즉시 임시 토큰은 파기하고 완전한 **App JWT 로그인 토큰**으로 승격시켜 발급(Issue)해 줍니다.

### 2) 비밀번호 검증 및 예외(NULL) 처리
- **비밀번호 검증 로직 (추가 적용됨):** 일반 가입 시 클라이언트(프론트엔드)에서 `6자리 이상 문자+숫자+특수문자` 조합인지 실시간 정규식(Regex)으로 엄격하게 검증하며, 일치 확인(PASSWORD 확인) 입력란을 통과해야만 백엔드로 전송됩니다.
- 반면 카카오/네이버 가입자는 데이터베이스 `PASSWORD` 컬럼에 `NULL` 값이 들어갈 수 있도록 예외처리되어 소셜 가입이 가능해야 합니다.

### 3) 독립적인 점유 인증 (`AUTH_VERIFY_YN`) 및 SMS 발송 아키텍처 전략
회원 가입의 가장 핵심적인 절차인 휴대폰 본인 인증(문자 OTP 전송)은 비용 효율성 및 개발 속도를 극대화하기 위해 **구글 클라우드 생태계를 활용한 단계적(Phase 1 → Phase 2) 스케일업 전략**을 채택합니다.

**[Phase 1] 런칭 및 초기 운영 코어: Firebase Authentication (파이어베이스 인증)**
- **도입 개요:** SMS 난수 생성, 문자 전송, 검증, 그리고 악의적인 매크로 전송을 차단하는 보안 장치(reCAPTCHA)를 프론트엔드에서 즉시 구현하기 위해 Google의 `firebase` Auth 모듈을 연동합니다.
- **도입 사유:** 매월 최대 10,000건의 문자를 **완전 무료**로 발송할 수 있어 초기 서비스 운영 비용 절감에 가장 유리하며 구현 난이도가 낮습니다. 백엔드는 복잡한 비밀 키 보관이나 Redis 인증번호 타이머를 관리할 필요 없이 프론트-구글 간의 인증 결과를 신뢰(Trust)하여 저장만 하면 됩니다.

**[Phase 2] 엔터프라이즈(B2B) 성장 확장: Google Cloud Identity Platform (GCIP)**
- **도입 개요 및 전환 방법:** 서비스가 커져 다중 접속 관리(Multi-Tenancy)나 기업용 최고 수준의 보안 준수(SLA, MFA 등)가 필요해질 때, Firebase 콘솔 화면에서 원클릭으로 플랫폼을 업그레이드합니다.
- **도입 사유:** 코드를 처음부터 다시 짜거나 DB 스키마(`TB_USER` 등)를 뜯어고치지 않고도, 백그라운드 엔진만 엔터프라이즈급 API인 GCIP로 무중단 격상(Migration)시킬 수 있는 플러그형 인프라로 대비되어 있습니다.

**[백엔드 데이터 처리 로직]**
- 회원 가입 [최종 완료 버튼]을 누르기 이전 단계에서, 앱/웹 프론트엔드가 위 구글 인증 시스템을 통해 휴대폰 점유 인증을 무사히 통과했는지 먼저 검사합니다.
- 정상적으로 문자와 숫자가 확인되어 검증 완료 토큰을 가진 세션 건에 한해서만 `TB_USER` INSERT 시 `AUTH_VERIFY_YN = 'Y'` 로 저장 조치(Commit)합니다.

### 4) 회원 탈퇴 및 상태 파악 (`USER_STAT_CD`)
- React 화면에서 회원이 탈퇴를 요청했을 때, 실제 DB Row를 `DELETE`하는 것이 아니라 백엔드가 `UPDATE TB_USER SET USER_STAT_CD='90'`으로 상태 값을 변경하는 설계 채택 필요 (법적 보관 기준 충족).

## 4. [최종 추천] busTaams 회원 가입 약관 노출 로직 및 도입 프로세스

회원 가입 플로우에서 약관을 보여주는 가장 이상적이고 현대적인 웹 UI 구조는 "PDF 파일 오픈" 방식을 지양하고, **프론트엔드 내에서 텍스트 기반 네이티브 팝업 스크롤링**을 적용하는 것입니다. 이를 기반으로 다음과 같은 도입 프로세스를 확립했습니다.

### 1) 약관 목록 구성 및 렌더링 방식
모든 약관은 체크박스 우측에 `[보기]` 버튼을 두어, 클릭 시 전체 창 이동이나 외부 다운로드 없이 회원 가입 모달 위에 "2중 자식 모달(Sub-Modal)" 형태로 약관 텍스트가 표시되게 구현합니다. 현재 회원가입 페이지에는 다음과 같은 권한 조건부 렌더링이 선행되어야 합니다.

- **기본 약관 (소비자 & 기사님 공통 노출)**
  1. 통합이용약관 (필수)
  2. 개인정보 처리방침 (필수)
  3. 위치정보 이용약관 (필수)
  4. 마케팅정보 수신 동의서 (선택)
- **파트너 전용 약관 (권한 분기)**
  5. `파트너(기사님) 입점 계약서` (필수) 
  *※ 회원 가입 모달 상단의 [소비자 / 기사님] 스위처에서 '기사님'을 선택했을 때만 동적으로 노출됩니다.*

### 2) 단계적 시스템 고도화 프로세스 정리
- **[1단계 (현재 MVP 방식) 정적 파일 연동]:** 약관 내용 전문을 `.md` 파일이나 프론트엔드의 `termsData.js` 객체 내 하드코딩된 텍스트로 보관합니다. `[보기]` 클릭 시 이 정적 데이터를 React가 Sub-Modal로 띄웁니다.
- **[2단계 (향후 목표) DB 연동]:** 추후 `busTaams_admin` 관리자 페이지가 구축되면 게시판 형태로 약관 내용을 DB 테이블에서 통제합니다. 프론트엔드에서는 회원 진입 시 최신 버전의 약관 DB를 API로 호출해 렌더링하므로 무중단 텍스트 배포가 가능합니다.

## 5. 추가 UI/UX 고도화 리포트
회원 가입 모달(`SignUpModal`)의 사용자 편의성 및 보안성을 극대화하기 위해 다음과 같은 세부 기능들이 프론트엔드 코드(`App.jsx`)에 최우선적으로 추가 반영되었습니다.

### 1) 휴대폰 본인 인증: 통신사 선택 드롭다운
- 사용자가 직접 통신사를 고를 수 있도록, `SKT, KT, LG U+` 및 각 `알뜰폰(MVNO)` 통신사 옵션을 담은 Select Box를 직관적으로 전화번호 입력란 바로 좌측에 배치했습니다. 

### 2) 전자 서명(사인) 및 약관 동의 락(Lock) 메커니즘
- 텍스트 입력 방식을 탈피하여, 사용자가 직접 마우스나 터치로 사인할 수 있는 **HTML5 Canvas 기반 서명 패드(`react-signature-canvas`)** 프론트엔드 모듈을 신설했습니다.
- 사용자가 그린 서명 데이터는 **PNG Data URL 포맷**으로 즉각 변환되어 State 객체에 저장되며 관리됩니다.
- 입력 중 **"지우기 (다시 쓰기)"** 버튼을 통해 언제든 캔버스를 초기화하고 처음부터 다시 서명할 수 있는 편의 기능이 내장되어 있습니다.
- **안전장치(Guard Logic):** 통합이용약관, 개인정보수집 등 '(필수)'가 붙은 약관이 모두 체크될 때까지만 캔버스 패드 영역이 `disabled` 상태로 단단히 덮여 잠깁니다. 기사님의 경우 '파트너(기사님) 입점 계약서'까지 모두 체크해야 캔버스가 활성화되어 서명이 가능해지는 동적 Guard 로직이 설계되었습니다.

### 3) 가입 완료 버튼 디자인 통일
- 최종 승인 역할을 하는 **'회원 가입 완료하기'** 버튼을 레퍼런스 이미지와 완벽하게 동일하도록, 직각을 없앤 완전한 둥근 알약 형태(`rounded-full`) 및 청록색 솔리드 전용 배경(`bg-[#007E69]`)으로 CSS 디자인을 표준화했습니다.

## 6. 일반 회원 가입(비소셜) 필수 예방 수칙 (4대 디테일)

추후 도입될 소셜 로그인을 배제하고 당장 **일반 이메일/비밀번호 가입 프로세스**를 우선 구축할 때, 프론트/백엔드/DB 파트간 필수적으로 점검해야 할 4가지 디테일은 다음과 같습니다.

### 1) 실명, 통신사 선택 및 필수 항목 처리
- **화면(UI):** 이메일(ID), Password 창 외에 사용자의 실명과 휴대폰 인증을 보조할 **통신사 선택 드롭다운(SKT, KT, LGU+, 알뜰폰 등)** 이 반드시 폼에 존재해야 인증 실패 확률을 줄일 수 있습니다.
- 또한 필수 약관 체크 외에도 프로모션 알람 목적의 **마케팅 정보 수신 동의(선택)** 박스를 노출하고 그 체크 상태값을 DB의 `MKT_AGREE_YN` 컬럼으로 전송해야 합니다.

### 2) 가입자 유형(소비자 vs 기사님) 동적 스위칭
- **화면(UI):** 이 가입자가 버스 대절 일반 예약자(`CONSUMER`)인지, 파트너 기사님(`DRIVER`)인지 명확하게 구분할 수 있는 탭(Tab) 또는 스위치 컴포넌트가 최상단에 배치되어야 합니다.
- 기사님(`DRIVER`)을 선택하면, 약관 동의 리스트에 **'기사님 파트너 입점 계약서'** 항목이 하나 더 추가(Dynamic Rendering)되어야 정상입니다.

### 3) 서명 데이터(PNG) 구글 클라우드 스토리지(GCS) 적재 우선 처리
- **프론트엔드:** 사용자가 캔버스(Canvas)에 그린 서명을 Base64 등의 규격으로 백엔드 API에 넘겨줍니다. 
- **백엔드:** 이 서명 데이터를 전달받은 서버는 DB부터 켜지 않고 제일 먼저 서명을 `.png` 파일로 변환합니다. 이후 구글 클라우드 보안 스토리지 버킷인 `gs://bustaams-secure-data/signatures/` 경로에 파일 업로드를 수행하고 구글로부터 원본 파일 식별자(UUID)를 반환받습니다.

### 4) 클라우드-DB 매핑(Mappings) 기반의 서명 이중 관리 (데이터 무결성)
- **DB 저장 로직:** 구글 클라우드 버킷은 단순한 그림 파일 창고용입니다. 이 서명이 법적 효력을 발휘하려면, 백엔드가 방금 전 구글 버킷에서 발급받은 그림 파일의 UUID와 경로를 꺼내서, **`TB_USER_TERMS_HIST`** 테이블에 INSERT(저장) 해야 합니다.
- 물리적 클라우드 객체는 `TB_FILE_MASTER`에 보관되고, 논리적 증거(회원의 가입 정보, 동의한 약관 ID 등)는 `TB_USER_TERMS_HIST`에 엮임으로써 "누가, 언제, 어떤 약관 버전에 동의하며, 버킷 어디에 있는 서명 파일을 남겼는지"를 시스템이 이중으로 보호 증명하게 됩니다.

### 5) 휴대폰 인증 서버리스 검증 및 요금제/과금 (Firebase Admin)
- **프론트 연동:** `App.jsx` 내부 `firebaseConfig`를 통해 구글 망에서 직접 카카오/KT/SKT 문자를 브라우저로 발송합니다.
- **백엔드 무결성 검증:** 백엔드 서버는 `busTaams_server/firebase-admin-key.json` (조직 정책을 풀고 실제 발급 완료된 Admin 비공개 권한 키)를 물고 실행됩니다. 프론트가 뱉어낸 인증 토큰을 서버리스로 교차 검증(`verifyIdToken`)하여, 조작되지 않은 정보만 회원가입 DB 처리 사이클로 넘깁니다. 
- **과금 이슈 점검:** SMS 문자 발송은 매달 10,000건까지 전액 무료이며, 백엔드의 토큰 교차 검증 로직은 횟수 제한 없이 100% 무료이므로 서버 테스트 시 "요금 폭탄"은 발생하지 않습니다.
- **주의 사항:** `firebase-admin-key.json`은 절대 소스코드 외부에 노출되선 안 되는 마스터 키이므로 Git 버전 관리대상에 올라가지 않도록 제외(`.gitignore` 추가)되어 있습니다.

## 7. 백엔드(Node.js) REST API 상세 설계 (일반 가입 기준)

앞서 정의된 프론트엔드 UI와 DB, 그리고 구글 클라우드 스토리지 로직을 모두 하나로 연결하기 통합하기 위해 백엔드(`busTaams_server`)가 구현해야 하는 3가지 핵심 API 명세입니다.

### API 1. 이메일 아이디 중복 검사
프론트엔드에서 회원이 이메일을 입력하고 넘어갈 때, 이미 가입된 계정인지 실시간으로 검증합니다.
- **Method & Endpoint:** `GET /api/auth/check-email`
- **Request (Query):** `?email=user@bustaams.com`
- **Response:**
  - `status: 200, isAvailable: true` (가입 가능)
  - `status: 409, message: "이미 사용 중인 이메일입니다."` (중복 발생 시)

### API 2. 시행 중인 최신 약관 목록 조회
회원가입 폼이 렌더링될 때 화면에 뿌려줄 필수/선택 약관 리스트를 DB(`TB_TERMS_MASTER`)에서 실시간으로 가져옵니다.
- **Method & Endpoint:** `GET /api/terms/active`
- **Response (JSON):**
  ```json
  {
    "status": 200,
    "data": [
      { "termsId": 1, "type": "SVC", "title": "통합이용약관", "isRequired": "Y" },
      { "termsId": 2, "type": "MKT", "title": "마케팅 동의", "isRequired": "N" },
      { "termsId": 3, "type": "DRIVER", "title": "기사님 규정", "isRequired": "Y" }
    ]
  }
  ```

### API 3. [핵심] 회원 가입 및 서명 최종 전송 (Transaction)
모든 검증을 마친 사용자가 **[회원 가입 완료하기]** 버튼을 눌렀을 때 호출되는 최종 관문 API입니다. DB 저장이 중간에 실패하면 GCS 파일 업로드도 롤백되도록 모두 **트랜잭션(Transaction)** 으로 묶어 처리해야 합니다.
- **Method & Endpoint:** `POST /api/auth/register`
- **Request (JSON Body):**
  ```json
  {
    "userId": "user@bustaams.com",
    "password": "EncryptedPassword@123",
    "userName": "홍길동",
    "phoneNo": "010-1234-5678",
    "userType": "CONSUMER",
    "firebaseIdToken": "eyJhbGciOiJSUzI1NiI...", 
    "mktAgreeYn": "Y",
    "signatureBase64": "data:image/png;base64,iVBORw0KGgoAAA...",
    "agreedTerms": [1, 2] 
  }
  ```
- **백엔드 내부 컨트롤러 로직 (순차 작동):**
  1. **[보안] 인증 토큰 검증:** `firebaseIdToken`을 Firebase Admin SDK로 풀어 실제 소유자의 전화번호와 일치하는지 백도어 유무를 교차 검증(`SMS_AUTH_YN = 'Y'` 자격 획득 여부 조회).
  2. **[식별자] UUID 생성:** 신규 회원의 고유 식별자인 `USER_UUID`를 시스템에서 생성 (Binary 16).
  3. **[보안] 암호화 처리:** 
     - 비밀번호: `bcrypt` 알고리즘으로 단방향 암호화 처리.
     - 이메일, 성명, 전화번호: `AES-256-GCM` 알고리즘으로 양방향 암호화 처리.
  4. **[스토리지] 서명 사진 적재:** `signatureBase64` 스트링을 버퍼로 변환하여 구글 클라우드 `signatures` 버킷에 업로드. 성공 시 `FILE_UUID` 확보를 위해 `TB_FILE_MASTER`에 선저장.
  5. **[DB 트랜잭션 시작] TB_USER INSERT:** 생성된 `USER_UUID`, 암호화된 개인정보, `SMS_AUTH_YN='Y'` 를 넣고 유저 프로필 신규 생성.
  6. **[DB 트랜잭션 종료] TB_USER_TERMS_HIST 다중 INSERT:** 가입자가 보낸 `agreedTerms` 배열을 반복하여, 각 약관 번호(`TERMS_ID`)마다 `SIGN_FILE_UUID`와 `USER_UUID`를 매핑하여 영구 이력을 남기고 트랜잭션 Commit.
- **Response:**
  - `status: 201, message: "회원가입이 성공적으로 완료되었습니다."` (정상 로그인 처리를 위한 JWT 토큰을 이어서 발급 가능)

## 8. 회원가입 중복 검증 로직 (Duplicate Validation)

회원 가입 시 데이터 무결성 보호를 위해 다음과 같은 중복 검증 체계가 적용되어 있습니다.

### 8-1. 전체 검증 항목 현황

| 검증 항목 | 검증 위치 | 검증 시점 | 처리 방식 |
|---|---|---|---|
| **이메일 중복** | 프론트 → 백엔드 API | 이메일 입력 후 포커스 이탈(blur) | API 조회 후 인라인 에러 표시 |
| **전화번호 중복** | 프론트 → 백엔드 API | SMS 인증번호 확인 완료 즉시 | API 조회 후 인라인 에러 표시 |
| **비밀번호 확인** | 프론트 | 가입 버튼 클릭 시 | `password !== passwordConfirm` 비교 |
| **필수항목 누락** | 프론트 | 가입 버튼 클릭 시 | 누락 목록 alert |
| **이메일 최종 중복** | 백엔드 DB | register API 처리 시 | `ER_DUP_ENTRY` → 409 응답 |

---

### 8-2. 백엔드 API 명세

#### API 1. 이메일 중복 검사 (`GET /api/auth/check-email`)
- **호출 시점:** 이메일 입력란에서 포커스가 이탈(blur)될 때 자동 호출
- **처리 방식:** DB 전체 USER_ID 조회 → AES-256-GCM 복호화 후 문자열 비교 (암호화 저장 대응)
- **Response:** `200 { isAvailable: true }` / `409 { message: "이미 사용 중인 이메일입니다." }`

#### API 1-2. 전화번호 중복 검사 (`GET /api/auth/check-phone`) ← **신규 추가**
- **호출 시점:** Firebase SMS 인증번호 확인 완료 즉시 자동 호출
- **처리 방식:** DB 전체 PHONE_NO 조회 → AES-256-GCM 복호화 후 문자열 비교 (암호화 저장 대응)
- **Response:** `200 { isAvailable: true }` / `409 { message: "이미 가입된 휴대폰 번호입니다." }`

---

### 8-3. 프론트엔드 검증 흐름도

```
[이메일 입력] → (blur) → GET /api/auth/check-email
                              ↓ 중복이면 인라인 에러 표시 (빨간 테두리 + 메시지)

[전화번호 입력] → [인증전송] → [인증번호 입력] → [확인] 
                                                    → GET /api/auth/check-phone
                                                         ↓ 중복이면 alert + 인증 취소
                                                         ↓ 정상이면 isPhoneVerified = true

[가입 버튼 클릭]
  → (1) 필수항목 누락 검사 (이름·이메일·비밀번호·전화인증·약관동의·서명)
  → (2) 비밀번호 === 비밀번호확인 검사
  → (3) emailError || phoneError 유무 최종 확인
  → (4) POST /api/auth/register (DB INSERT)
         └→ ER_DUP_ENTRY 발생 시 409 응답으로 최후 방어
```

---

### 8-4. 암호화 필드 중복 검사 주의사항

> ⚠️ **중요:** 이메일과 전화번호는 AES-256-GCM으로 암호화(IV가 매번 다름)되어 저장되므로, 동일한 값이라도 DB에 저장된 암호문은 매번 다릅니다. 따라서 `WHERE USER_ID = ?` 방식의 직접 비교가 불가능하며, **전체 조회 후 복호화 비교** 방식을 사용합니다.
>
> 회원 수가 많아지면 성능 이슈가 발생할 수 있으므로, 향후에는 **결정적 암호화(Deterministic Encryption)** 또는 **해시 인덱스 컬럼 별도 관리** 방식으로 고도화를 권장합니다.
