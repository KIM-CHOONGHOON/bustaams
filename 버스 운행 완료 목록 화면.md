# 버스 운행 완료 목록 화면 (BusOperationCompletionList)

## 0. 스키마·식별자 기준 (`ARCHITECTURE.md`)

- **공식 DDL:** [`ARCHITECTURE.md`](./ARCHITECTURE.md)의 `CREATE TABLE` — 본 화면과 직접 연관되는 테이블은 **`TB_BUS_RESERVATION`**, **`TB_AUCTION_REQ`**, **`TB_AUCTION_REQ_BUS`**, **`TB_USER`** 등이다.
- **UUID → 비즈니스 ID:** 레거시는 `RES_UUID` / `REQ_UUID` / `REQ_BUS_UUID` / `DRIVER_UUID`(BINARY) 등을 쓸 수 있다. **목표 스키마**는 문자열 키 **`RES_ID`**, **`REQ_ID`**, **`REQ_BUS_ID`**, 상태 **`DATA_STAT`**(`'DONE'` = 운행 완료), 기사 **`DRIVER_ID`**(예약 행에 저장된 `TB_USER.USER_ID`와 동일 의미)이다.
- **ID 채번:** [`ARCHITECTURE.md`](./ARCHITECTURE.md) **「비즈니스 식별자(ID) 채번 규칙」** — `RES_ID`, `REQ_ID`, `REQ_BUS_ID` 등 **`VARCHAR(n)` 순번형** 컬럼은 해당 **`n`** 자리 **앞자리 0 채움**(`padStart`). **`TB_USER.USER_ID`**는 가입에서 결정되는 값으로 **순번 0채움 규칙의 예외**일 수 있다.
- **REST·프론트 명명 권장:** 쿼리·JSON·props는 DB 컬럼과 맞추어 **`driverId`**(=`TB_USER.USER_ID`), **`resId`**(=`RES_ID`), **`reqId`**(=`REQ_ID`), **`reqBusId`**(=`REQ_BUS_ID`)를 쓴다. 레거시 호환으로 **`driverUuid`**, **`resUuid`**, **`reqUuid`**, **`reqBusUuid`** 이름이 남아 있을 수 있으나, **값이 반드시 UUID 형식은 아니다**(대개 0채움 ID 문자열).
- **타입 주의:** `ARCHITECTURE.md` DDL에서 `TB_BUS_RESERVATION.DRIVER_ID` 등이 `varchar(10)`으로 적혀 있어도, **`TB_USER.USER_ID`가 `varchar(255)`** 이면 실제 DB는 마이그레이션으로 길이를 맞춘다. 조회·비교 시에는 **운영 DDL의 실제 길이**를 따른다.

### 2-1. UI (HTML 준수 범위)

- 제목: **버스 운행 완료 목록**
- 입력: **From**, **To** (`type="date"`, `YYYY-MM-DD`)
- 버튼: **조회**
- 출력: 테이블 — No, 여정명, 출발지, 도착지(`END_ADDR`), **요청 차량**(`TB_AUCTION_REQ_BUS` 요약), **종료일**(`END_DT`의 일자 `YYYY-MM-DD`), 종료 일시(`END_DT` 전체), 입찰가(`DRIVER_BIDDING_PRICE`)
- 모달 열림 시 From/To 기본값: **당월 1일 ~ 오늘**

### 2-1-1. 조회 기간 제한 (최대 1년)

| 항목 | 내용 |
|------|------|
| **규칙** | `From`·`To`의 **일수 차이**가 **365일을 넘을 수 없음** (`To − From` ≤ 365일, UTC 일 단위). |
| **프론트** | `To` 입력에 `min=From`, `max=From+365일`; 범위를 벗어나면 값 보정 또는 안내 모달. **조회**·최초 자동 조회 전 동일 검증. |
| **안내 모달** | 조건 위반 시 문구 **`조회 범위는 1년 이내만 가능합니다.`** (또는 `종료일(To)은 시작일(From) 이후여야 합니다.`) — **확인** 클릭 시 모달만 닫힘 (`z-[110]`). |

### 2-2. 모달 호출 시 조회·표시 순서 (END_DT ↔ From/To)

1. 모달이 **열리면** From·To는 **항상 `YYYY-MM-DD`** 로 맞춘 뒤(프론트 `toYYYYMMDD` 정규화), 기본값 **당월 1일 ~ 오늘**이 입력란에 설정된다.
2. **즉시** `GET /api/bus-operation-completion-list` 를 호출한다(`driverId`·`from`, `to` 쿼리; 레거시는 `driverUuid`).
3. 백엔드는 `TB_AUCTION_REQ`.`END_DT`를 **`DATE(END_DT)`** 로 잘라, **`from` ≤ `DATE(END_DT)` ≤ `to`** 인 행만 추출한다(시각은 비교에서 제외, **일 단위**).
4. 추출 결과를 **`END_DT` 내림차순**으로 정렬해 모달 테이블에 출력한다.
5. 각 행의 **종료일** 열에는 `END_DT`에서 뽑은 **`YYYY-MM-DD`**(`endDtDate`)를, **종료 일시** 열에는 원본 일시를 표시한다.

## 3. 백엔드·REST API

| 항목 | 내용 |
|------|------|
| **메서드** | `GET` |
| **경로** | `/api/bus-operation-completion-list` |
| **구현** | `busTaams_server/server.js` (등록) · `busTaams_server/routes/busOperationCompletionList.js` (핸들러) |

### 3-1. 쿼리 파라미터

| 이름 | 필수 | 설명 |
|------|------|------|
| `driverId` | O (권장) | 로그인 기사 **`TB_USER.USER_ID`** — DB `TB_BUS_RESERVATION.DRIVER_ID`와 직접 비교 가능할 때 사용 |
| `driverUuid` | O (레거시) | 로그인 기사 **`BIN_TO_UUID(USER_UUID)`** 등 구 스키마 세션 값 — 서버가 `TB_USER`에서 `USER_ID`로 해석해 매칭 |
| `from` | O | 시작일 `YYYY-MM-DD` |
| `to` | O | 종료일 `YYYY-MM-DD` |

> 구현체는 **`driverId` 또는 `driverUuid` 중 하나**를 받도록 정리하는 것이 좋다. 둘 다 올 때는 **`driverId` 우선** 권장.

### 3-1-1. 기간 검증 (API·DB 조회 전)

| 조건 | HTTP | 응답 `error` (JSON) |
|------|------|----------------------|
| `from`·`to` 형식 오류 | `400` | `from, to는 YYYY-MM-DD 형식이어야 합니다.` 등 |
| `To`가 `From`보다 이전(일 단위) | `400` | `종료일(To)은 시작일(From) 이후여야 합니다.` |
| `To − From` &gt; **365일** | `400` | **`조회 범위는 1년 이내만 가능합니다.`** |

### 3-2. 조회 조건 (업무)

**구현:** `routes/busOperationCompletionList.js` — DB에 레거시 컬럼(`RES_UUID`, `RES_STAT`, …)이 있으면 **레거시 SQL**을 쓰고, `ER_BAD_FIELD_ERROR`(1054)이면 **`ARCHITECTURE.md` 스키마용 SQL**로 재시도한다.

**레거시**

1. `TB_BUS_RESERVATION`.`RES_STAT` = `'DONE'`
2. `LOWER(BIN_TO_UUID(res.DRIVER_UUID))` = 요청 `driverUuid`(소문자 정규화)
3. `TB_AUCTION_REQ`와 **`REQ_UUID`**로 조인
4. **요청 차량:** 각 `REQ_UUID`에 대해 `TB_AUCTION_REQ_BUS`를 `REQ_UUID IN (UUID_TO_BIN(?), …)` 로 조회 (`REG_DT`, `REQ_BUS_UUID` 순).

**ARCHITECTURE (`DATA_STAT`, `RES_ID` / `REQ_ID` / `REQ_BUS_ID`)**

1. `TB_BUS_RESERVATION`.`DATA_STAT` = `'DONE'`
2. `res.DRIVER_ID` = 로그인 기사의 **`TB_USER.USER_ID`**  
   - **권장(정합):** 쿼리 파라미터 `driverId`를 그대로 `res.DRIVER_ID`와 비교.  
   - **하이브리드:** `USER_UUID = UUID_TO_BIN(driverUuid)` 인 행의 `COALESCE(NULLIF(TRIM(USER_SEQ_ID),''), USER_ID)` 와 비교(기존 `routes/busOperationCompletionList.js` 패턴).
3. `TB_AUCTION_REQ`와 **`REQ_ID`**로 조인 (`r.REQ_ID = res.REQ_ID`).
4. **요청 차량:** 각 `REQ_ID`에 대해 `TB_AUCTION_REQ_BUS`를 `REQ_ID IN (?)` 로 조회. [`ARCHITECTURE.md`](./ARCHITECTURE.md) DDL에는 **`REQ_BUS_CNT`/`REQ_AMT`가 없을 수 있음** — **`TOLLS_AMT`+`FUEL_COST`** 로 금액 합산해 `reqAmtKrw`에 대응.

**공통**

- **기간(일 단위):** `DATE(TB_AUCTION_REQ.END_DT)` 가 `from`·`to` **포함 구간**. SQL: `DATE(r.END_DT) >= ? AND DATE(r.END_DT) <= ?`
- 응답 **`endDtDate`**: `DATE_FORMAT(r.END_DT, '%Y-%m-%d')`
- **정렬:** `r.END_DT DESC`; **`r.END_DT IS NOT NULL`**
- 응답 **`resId`** (레거시 키명 `resUuid`): 레거시는 `BIN_TO_UUID(RES_UUID)`, ARCH는 **`RES_ID`** (0채움 순번 문자열일 수 있음)
- 응답 **`reqId`** (레거시 `reqUuid`): 레거시는 `BIN_TO_UUID(REQ_UUID)`, ARCH는 **`REQ_ID`**
- 응답 **`auctionReqBuses[].reqBusId`** (레거시 `reqBusUuid`): ARCH는 **`REQ_BUS_ID`**

### 3-3. 오류 응답 (기간 외)

## 6. 문제 해결 (404)

- 응답 본문에 **`Cannot GET /api/bus-operation-completion-list`** 가 보이면, **Node(Express) 프로세스가 라우트 추가 이전 코드로 떠 있는 것**입니다. `busTaams_server`에서 **서버를 한 번 종료한 뒤 다시 실행**하세요 (`node server.js` 또는 `nodemon` 등).
- 구현 위치: `server.js` 상단에서 `require('./routes/busOperationCompletionList')(pool, app)` 로 등록됩니다(`routes/busOperationCompletionList.js` 내 `app.get('/api/bus-operation-completion-list', …)`).
- **`VITE_API_BASE_URL`** 이 실제 API 포트와 다르면 다른 호스트로 요청이 가 404가 날 수 있습니다. 비우면 개발 시 **상대 경로 `/api` → Vite 프록시(8080)** 로 동일 오리진 요청이 갑니다.
- 개발 모드에서는 요청 URL이 브라우저 콘솔에 `console.warn` 으로 남습니다.

## 7. `RES_STAT = 'DONE'` 인데 모달에 1건만 나오는 경우

`TB_BUS_RESERVATION` 에 DONE이 **여러 건** 있어도, 아래를 **모두** 만족해야 목록에 나옵니다. 하나라도 빠지면 **제외**됩니다.

| 점검 | 설명 |
|------|------|
| **기간(From~To)** | 필터는 **`TB_AUCTION_REQ.END_DT`의 날짜**입니다. 모달 기본값은 **당월 1일~오늘**이라, 견적의 **도착 예정일 `END_DT`** 가 이 구간 **밖**이면 나오지 않습니다. `REG_DT`·`MOD_DT`·`CONFIRM_DT`는 **이 API에서 사용하지 않습니다** (`CONFIRM_DT`가 NULL이어도 무관). |
| **`END_DT` NULL** | `r.END_DT IS NOT NULL` 조건으로, 마스터에 도착 일시가 없으면 **제외**됩니다. |
| **조인** | `INNER JOIN TB_AUCTION_REQ` 이므로 **레거시 `REQ_UUID` / ARCH `REQ_ID`** 에 대응하는 견적 행이 없으면 **제외**됩니다. |
| **기사** | **레거시:** `res.DRIVER_UUID` = 로그인 기사 `UUID_TO_BIN`. **ARCH:** `res.DRIVER_ID` = 해당 기사 **`TB_USER.USER_ID`** (또는 하이브리드 시 `USER_SEQ_ID`와 동기화된 값). |

**확인용 SQL (레거시)**

```sql
SELECT BIN_TO_UUID(res.RES_UUID), BIN_TO_UUID(res.REQ_UUID),
       res.RES_STAT, res.DRIVER_UUID,
       r.END_DT, DATE(r.END_DT) AS end_day
FROM TB_BUS_RESERVATION res
LEFT JOIN TB_AUCTION_REQ r ON r.REQ_UUID = res.REQ_UUID
WHERE res.RES_STAT = 'DONE'
  AND LOWER(BIN_TO_UUID(res.DRIVER_UUID)) = LOWER('로그인-기사-UUID-문자열');
```

**확인용 SQL (ARCHITECTURE — `driverId` 직접 바인딩 권장)**

```sql
SELECT res.RES_ID, res.REQ_ID, res.DATA_STAT, res.DRIVER_ID,
       r.END_DT, DATE(r.END_DT) AS end_day
FROM TB_BUS_RESERVATION res
LEFT JOIN TB_AUCTION_REQ r ON r.REQ_ID = res.REQ_ID
WHERE res.DATA_STAT = 'DONE'
  AND res.DRIVER_ID = '로그인-기사-USER_ID';
```

**하이브리드(DB에 `USER_UUID` 컬럼이 남아 있을 때)**

```sql
-- 위 WHERE 절만 아래로 교체 가능
AND res.DRIVER_ID = (
  SELECT COALESCE(NULLIF(TRIM(USER_SEQ_ID),''), USER_ID)
  FROM TB_USER WHERE USER_UUID = UUID_TO_BIN('로그인-기사-UUID-문자열') LIMIT 1);
```

한 건은 `END_DT`가 From~To 안에 들어가고, 다른 한 건은 **다른 달·다른 날**이거나 **`END_DT` NULL**일 가능성이 큽니다. From/To를 넓히거나, 해당 견적의 `END_DT`를 확인하면 됩니다.

## 9. 라우트 등록 방식·로컬 검증 (수정 내용)

`app.use('/api', Router)` + `Router.get('/bus-operation-completion-list')` 조합은 환경에 따라 경로가 맞지 않아 **404**가 날 수 있어, 아래처럼 **전체 경로를 한 번에** 등록한다.

| 항목 | 내용 |
|------|------|
| **파일** | `routes/busOperationCompletionList.js` |
| **등록** | `app.get('/api/bus-operation-completion-list', …)` |
| **호출** | `server.js` 상단(글로벌 로거 다음)에서 `registerBusOperationCompletionList(pool, app)` |

로컬에서 Express 앱에 위 핸들러만 붙여 `GET /api/bus-operation-completion-list?...`를 호출하면 **`200`** / `{"total":0,"items":[]}` 형태 응답이 나오는지 확인할 수 있다(유효한 `driverUuid`·DB 연결 시 데이터는 달라질 수 있음).
