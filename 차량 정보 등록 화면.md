# 차량 정보 등록 화면

기사 웹에서 **바로가기 → 「버스 정보」** 로 열리는 **`BusInformationSetup`** 모달과 연동되는 화면 정의입니다. 모달은 **열면** `GET /api/driver/bus` 로 기존 차량이 있으면 폼·서류·사진을 채우고, 없으면 신규 등록으로 동작한다. 저장 시 **신규**는 `POST /api/driver/bus`, **수정**은 `PATCH /api/driver/bus` + (필요 시) `PATCH /api/driver/bus/documents` · `PATCH /api/driver/bus/photos` 를 사용한다.

---

## 1. 목적

- 기사가 보유 차량의 **식별 정보·서비스 등급·편의 옵션·점검·보험 일자** 등을 입력한다.
- 저장 시 서버가 **`TB_DRIVER_BUS`**(또는 동등 스키마)에 **INSERT** 할 수 있도록 요청 본문을 구성한다.

---

## 2. 테이블 정의 (대상 스키마)

아래는 설계 기준 DDL입니다. `BUS_ID INT BINARY(16)` 은 일반적으로 **`BINARY(16)` 단일 PK** 또는 `BINARY(16) NOT NULL` 로 정의하는 것이 맞습니다(문서의 `INT` 는 오타로 보정).

```sql
CREATE TABLE TB_DRIVER_BUS (
    BUS_ID BINARY(16) NOT NULL PRIMARY KEY COMMENT '버스 고유 ID',
    USER_UUID BINARY(16) NOT NULL COMMENT '소유 기사 식별자 (TB_USER 참조)',

    -- 차량 기본 정보
    BUS_NO VARCHAR(20) NOT NULL COMMENT '차량 번호 (예: 서울70아1234)',
    BUS_MODEL VARCHAR(50) COMMENT '차량 모델명 (예: 유니버스, 그랜버드)',
    MANUFACTURE_YEAR YEAR COMMENT '차량 연식',

    BUS_SIZE ENUM('LARGE', 'MEDIUM', 'SMALL') NOT NULL DEFAULT 'LARGE'
        COMMENT '차량 크기 (LARGE:대형 12m, MEDIUM:중형 7-9m, SMALL:소형/미니)',

    BUS_GRADE ENUM('GENERAL', 'DELUXE', 'PREMIUM', 'VIP_MINI') NOT NULL DEFAULT 'GENERAL'
        COMMENT '서비스 등급 (GENERAL:45인승일반, DELUXE:28인승우등, PREMIUM:21인승, VIP_MINI:의전용)',

    TOTAL_SEATS INT NOT NULL DEFAULT 45 COMMENT '실제 총 좌석 수',

    HAS_REFRIGERATOR ENUM('Y', 'N') DEFAULT 'N' COMMENT '냉장고 보유 여부',
    HAS_WIFI         ENUM('Y', 'N') DEFAULT 'N' COMMENT '와이파이 제공 여부',
    HAS_USB_PORT     ENUM('Y', 'N') DEFAULT 'N' COMMENT '좌석별 USB 포트 여부',
    HAS_MONITOR      ENUM('Y', 'N') DEFAULT 'N' COMMENT 'TV/모니터 보유 여부',
    HAS_TABLE_SEAT   ENUM('Y', 'N') DEFAULT 'N' COMMENT '살롱/테이블석 여부',

    -- ADAS(안전운전 보조) — 화면의 「최신 ADAS」 토글과 매핑 (스키마에 컬럼 추가 시)
    HAS_ADAS         ENUM('Y', 'N') DEFAULT 'N' COMMENT 'ADAS(긴급제동·차선이탈 등) 장착 여부',

    INSURANCE_RENEWAL_DT DATE COMMENT '보험 갱신 예정일',
    LAST_INSPECTION_DT   DATE COMMENT '최근 차량 점검일',

    REG_DT DATETIME DEFAULT CURRENT_TIMESTAMP COMMENT '등록 일시',
    REG_ID VARCHAR(30) COMMENT '등록자 ID',
    MOD_DT DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '수정 일시',
    MOD_ID VARCHAR(30) COMMENT '수정자 ID',

    CONSTRAINT FK_BUS_DRIVER FOREIGN KEY (USER_UUID) REFERENCES TB_USER (USER_UUID)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='기사 보유 차량 및 상세 옵션 관리';
```

### 2-1. 현재 구현(`busTaams_server/server.js`)과의 차이

| 항목 | 설계 DDL | 현재 서버(`CREATE TABLE IF NOT EXISTS`) |
|------|-----------|----------------------------------------|
| PK | `BUS_ID` | `BUS_UUID BINARY(16)` |
| 차량번호 | `BUS_NO` | `VEHICLE_NO` |
| 모델 | `BUS_MODEL` | `MODEL_NM` |
| 연식 | `MANUFACTURE_YEAR YEAR` | `MANUFACTURE_YEAR VARCHAR(10)` |
| 등급 | `BUS_GRADE` ENUM | `SERVICE_CLASS VARCHAR(50)` |
| 크기·좌석 | `BUS_SIZE`, `TOTAL_SEATS` | (미구현) |
| 편의옵션 | `HAS_*` ENUM | `AMENITIES JSON` (한 번에 저장) |
| 점검 | `LAST_INSPECTION_DT` | `LAST_INSPECT_DT` |
| 보험 | `INSURANCE_RENEWAL_DT` | `INSURANCE_EXP_DT` |
| 차량 사진(최대 8) | (설계 확장) | `VEHICLE_PHOTOS_JSON` — `FILE_UUID` 문자열 배열 |

**권장:** 운영 DB를 설계 DDL로 맞출 때는 **마이그레이션**으로 컬럼명·ENUM을 정리하고, 프론트는 `amenities` 객체를 **Y/N 컬럼으로 변환**하거나 `AMENITIES JSON`에 `adas`, `wifi`, … 를 포함해 **하위 호환**으로 저장할 수 있다.

## 4. `TB_COMMON_CODE` (버스 종류)

버스 종류 목록은 **공통코드** 테이블에서 조회한다.

### 4-1. 조회 조건

```sql
SELECT DTL_CD, CD_NM_KO, CD_DESC
FROM TB_COMMON_CODE
WHERE GRP_CD = 'BUS_TYPE'
  AND USE_YN = 'Y'
ORDER BY DISP_ORD ASC, DTL_CD ASC;
```

| 컬럼 | 설명 |
|------|------|
| `GRP_CD` | 그룹코드 — 버스 종류는 **`BUS_TYPE`** (대문자, DB·쿼리와 동일) |
| `DTL_CD` | 상세코드 — 저장 시 `serviceClass` 로 전송 (예: `NORMAL_45`, `PRESTIGE_28`) |
| `CD_NM_KO` | 화면 버튼 **제목** (예: 일반 버스 (45석)) |
| `CD_DESC` | (선택) **부가 설명** — 예: 좌석 구성 안내. 구 DB에 `CD_DESC_KO` 만 있는 경우 API가 폴백 조회함 |
| `USE_YN` | `Y` 만 노출 |
| `DISP_ORD` | 정렬 순서 |

### 4-2. 서버가 생성하는 테이블 스키마 (참고)

`busTaams_server` 는 최초 조회 시 `TB_COMMON_CODE` 가 없으면 아래와 유사하게 생성하며, `BUS_TYPE` 이 비어 있으면 개발용 시드 데이터를 넣을 수 있다. 운영 DB에는 동일 스키마에 맞춰 데이터를 등록한다.

```sql
CREATE TABLE IF NOT EXISTS TB_COMMON_CODE (
    GRP_CD VARCHAR(50) NOT NULL,
    DTL_CD VARCHAR(50) NOT NULL,
    CD_NM_KO VARCHAR(200) NOT NULL DEFAULT '',
    CD_DESC VARCHAR(500) NULL,
    USE_YN CHAR(1) NOT NULL DEFAULT 'Y',
    DISP_ORD INT NOT NULL DEFAULT 0,
    PRIMARY KEY (GRP_CD, DTL_CD),
    KEY IDX_COMMON_GRP_USE_ORD (GRP_CD, USE_YN, DISP_ORD)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

## 5. API

### 5-1. 버스 종류 목록

- **GET** `/api/common-codes?grpCd=BUS_TYPE`
- **응답:** `{ items: [ { dtlCd, cdNmKo, cdDescKo }, ... ] }`
- 용도: 차량 정보 등록 모달 **02. 서비스 등급** 그리드 버튼 생성

### 5-2. 기사 소유 차량 조회(모달 기본값)

- **GET** `/api/driver/bus?userUuid=` (필수)
- **응답:** `{ bus: null }` — 등록 차량 없음(신규 등록 UI).  
  또는 `{ bus: { busUuid, vehicleNo, modelNm, manufactureYear, mileage, serviceClass, amenities, hasAdas, lastInspectDt, insuranceExpDt, bizRegFile, transLicFile, insCertFile, vehiclePhotoFileUuids } }`
  - `bizRegFile` / `transLicFile` / `insCertFile`: 각각 `{ fileUuid, orgFileNm, fileExt }` 또는 `null`
  - `vehiclePhotoFileUuids`: `FILE_UUID` 문자열 배열(순서대로, 최대 8)
- **프론트:** `BusInformationSetup` 마운트 시 호출해 입력란·버스 종류·어메니티·날짜를 채운다. 서류·차량 사진은 **`GET /api/driver/bus-documents/file?userUuid=&fileUuid=`** 로 스트리밍해 미리보기(이미지는 `object-contain`으로 업로드 영역 안에 맞춤, PDF는 아이콘+파일명).

### 5-3. 차량 등록

- **POST** `/api/driver/bus`
- **요청 본문(필드):** `userUuid`, `vehicleNo`, `modelNm`, `manufactureYear`, `mileage`, **`serviceClass`**(필수, `DTL_CD` 값), `amenities`, `lastInspectDt`, `insuranceExpDt`
- **차량 사진(선택):** `vehiclePhotos` — 배열, 최대 **8개** 요소. 각 요소 `{ base64, fileName }`(또는 `dataUrl`/`name`). **이미지** JPG·PNG·WebP, 각 최대 5MB. 순서대로 `VEHICLE_PHOTOS_JSON` 에 저장.
- **서류(선택, data URL base64):**
  - `businessLicenseBase64`, `businessLicenseFileName` → `Business_License` / `BIZ_REG_FILE_UUID`
  - `transportationLicenseBase64`, `transportationLicenseFileName` → `Transportation_Business_License` / `TRANS_LIC_FILE_UUID`
  - `insurancePolicyBase64`, `insurancePolicyFileName` → `Insurance_Policy` / `INS_CERT_FILE_UUID`
- **처리 순서:** `TB_DRIVER_BUS` INSERT(서류 FK는 NULL) → 각 첨부에 대해 GCS(`vehicle_docs/{YYYY-MM}/{FILE_UUID}.확장자`) → `TB_FILE_MASTER` INSERT → `TB_DRIVER_BUS_FILE_HIST` INSERT → `TB_DRIVER_BUS` 해당 컬럼 UPDATE.
- **성공:** `201`, `{ message, busUuid, fileUuids?, vehiclePhotoFileUuids? }` (각각 첨부가 있을 때만)

### 5-4. 차량 정보 수정(기본 필드)

- **PATCH** `/api/driver/bus`
- **본문:** `userUuid`, `busUuid`(필수), 변경할 필드만 또는 전체 — `vehicleNo`, `modelNm`, `manufactureYear`, `mileage`, `serviceClass`, `amenities`, `hasAdas`, `lastInspectDt`, `insuranceExpDt` 등.
- **성공:** `200`, `{ message }`

### 5-5. 서류 재업로드(이력 누적)

- **PATCH** `/api/driver/bus/documents`
- **본문:** `userUuid`, `busUuid`(필수), 위와 동일한 서류 필드 중 **보낸 항목만** 갱신(새 `FILE_UUID` + 이력 INSERT + FK UPDATE).
- **성공:** `200`, `{ message, fileUuids? }`

### 5-6. 차량 사진 재등록(이력 누적)

- **PATCH** `/api/driver/bus/photos`
- **본문:** `userUuid`, `busUuid`(필수), **`vehiclePhotos`** — 최대 8개. 각 요소는 (1) **기존 유지:** `{ fileUuid }` 만, (2) **신규 업로드:** `{ base64, fileName }` 또는 `dataUrl`/`name`. 보낸 **순서대로** 처리한 뒤 **`VEHICLE_PHOTOS_JSON`** 전체를 해당 목록으로 교체한다(빈 배열이면 사진 없음).
- **성공:** `200`, `{ message, vehiclePhotoFileUuids? }`

### 5-7. 서류 이력 조회(소유 기사만)

- **GET** `/api/driver/bus-documents/history?userUuid=&busUuid=`
- **응답:** `{ items: [ { fileUuid, fileCategory, regDt, orgFileNm, fileExt, fileSize }, ... ] }` (최신순)

### 5-8. 서류·차량 사진 원본 스트리밍(소유 기사만)

- **GET** `/api/driver/bus-documents/file?userUuid=&fileUuid=`
- **허용:** (1) **`TB_DRIVER_BUS_FILE_HIST`** 에 올라간 `FILE_UUID`(서류·`BUS_PHOTO` 차량 사진 모두). (2) **구버전 호환:** `VEHICLE_PHOTOS_JSON` 에만 있고 구 `Vehicle_Photo` 로 저장된 객체(이력 없음) — 동일 기사 소유 차량에 한함. 그 외 **403**.
- **응답:** GCS 객체 스트림 (`Content-Disposition: inline`, 원본 파일명 힌트)

### 5-9. 구현 메모 (`busTaams_server` / `busTaams_web`)

| 구분 | 내용 |
|------|------|
| JSON 용량 | 서류 base64 포함을 위해 `express.json` 한도 **50mb** |
| 이력 테이블 | 요청 시 `ensureTbDriverBusFileHistTable` 로 없으면 생성(운영 DB에 이미 있으면 그대로 사용) |
| 프론트 | `BusInformationSetup.jsx` — 모달 오픈 시 **`GET /api/driver/bus`** 로 기존 데이터 로드·미리보기. **신규:** `POST /api/driver/bus`. **수정:** `PATCH /api/driver/bus` → (새 서류만) `PATCH /api/driver/bus/documents` → `PATCH /api/driver/bus/photos`(기존 `fileUuid` + 신규 base64 혼합). 하단 제출 버튼은 **수정 모드**에서 「**차량 수정 완료**」, 신규에서 「차량 등록 완료」. 조회 중에는 「불러오는 중…」·오버레이 스피너로 제출을 막는다. |
| 차량 사진 DB | `ensureTbDriverBusVehiclePhotosColumn` — `TB_DRIVER_BUS.VEHICLE_PHOTOS_JSON` 없으면 `ALTER` 로 추가 |
| 차량 사진 이력 | `FILE_CATEGORY` = **`BUS_PHOTO`** 로 `TB_DRIVER_BUS_FILE_HIST` 에 매 장마다 INSERT |

## 6. UI — 02. 서비스 등급 (버스 종류)

- 상단 **버스 실사 이미지가 있는 대형 카드 UI는 사용하지 않는다.** (삭제됨)
- `GET /api/common-codes?grpCd=BUS_TYPE` 로 받은 항목만큼 **버튼**을 그린다.
- **레이아웃:** CSS 그리드 **`grid-cols-3`** — 한 줄에 **3개**, 항목이 더 많으면 **다음 줄**에 이어서 배치한다.
- **스타일:** 기존 「일반 버스」와 같은 형태의 직사각형 버튼(제목 + 부가설명).
- **선택 규칙:** **라디오와 동일** — 목록 중 **하나만** 선택 가능하다. 다른 버튼을 누르면 이전 선택이 해제되고 새 코드가 선택된다. (`serviceClass` = 선택된 행의 `DTL_CD`)

## 7. UI 동작 (편의 시설)

- **「제공 가능한 어메니티」** 는 각 항목을 **버튼형 토글**로 동작한다.
- **선택 시:** 해당 옵션이 켜진 것으로 보이도록 **체크 표시(체크박스)** 가 나타난다.
- **다시 클릭 시:** 선택 해제되며 **체크 표시가 사라진다** (미선택 상태).
- **순서:** **최신 ADAS** → 와이파이 → USB 충전 → 개별 스크린 → 냉장고 → 테이블 (와이파이 앞에 ADAS).

## 8. 검사 및 서류 — 파일 업로드 (`TB_FILE_MASTER`)

차량 정보 등록 화면 하단의 **사업자 등록증 / 운송사업 허가증 / 보험증권** 업로드 영역은, 실제 파일은 **Google Cloud Storage** 에 두고 메타데이터만 **`TB_FILE_MASTER`** 에 적재한다.

### 8-1. 컬럼 매핑 (`TB_FILE_MASTER`)

| 컬럼 | 저장 내용 |
|------|-----------|
| **FILE_UUID** | 서버에서 생성한 UUID. **PK**이며 GCS 객체 식별에도 동일 값을 사용한다. |
| **FILE_CATEGORY** | 아래 **§8-2**의 고정 코드 문자열 |
| **GCS_BUCKET_NM** | 업로드 대상 버킷명 (예: 환경·설정값) |
| **GCS_PATH** | 버킷 내 경로. **파일명(객체 키의 파일 부분)은 `FILE_UUID` 를 사용**한다. 구현 예: `vehicle_docs/{YYYY-MM}/{FILE_UUID}.pdf` — 확장자는 실제 파일과 동일 |
| **ORG_FILE_NM** | 사용자가 선택한 **원본 파일명** (표시·감사용) |
| **FILE_EXT** | 확장자만 저장 (예: `pdf`, `jpg`). 점(`.`) 제외 권장 |
| **FILE_SIZE** | 바이트 단위 파일 크기 |
| **REG_DT** | 등록 일시 |

**저장 순서(권장):** `FILE_UUID` 발급 → GCS에 바이너리 업로드(객체 이름에 `FILE_UUID` 반영) → 성공 시 `TB_FILE_MASTER` INSERT → 차량 행(`TB_DRIVER_BUS` 등)의 `BIZ_REG_FILE_UUID` 등에 동일 `FILE_UUID` 를 FK 로 연결.

### 8-2. `FILE_CATEGORY` 코드 (화면 ↔ DB 상수)

| 화면 라벨 | `FILE_CATEGORY` 값 |
|-----------|---------------------|
| 사업자 등록증 | `Business_License` |
| 운송사업 허가증 | `Transportation_Business_License` |
| 보험증권 | `Insurance_Policy` |
| 차량 사진(등록 화면 05) | **`BUS_PHOTO`** |

`Business_License` 등 세 값은 **차량 정보 등록 서류** 전용, **`BUS_PHOTO`** 는 **동일 화면의 차량 사진**에 사용한다. (기존 기사 프로필의 `DRIVER_PROFILE_PHOTO`, `BUS_QUAL_CERT` 등과 구분) 과거에 `Vehicle_Photo` 로 저장된 데이터가 있으면 ENUM/`VARCHAR` 마이그레이션 시 **`BUS_PHOTO`** 로 통일하는 것을 권장한다.

> 구현 시 서버 상수 예: `Business_License` · `Transportation_Business_License` · `Insurance_Policy` · `BUS_PHOTO`

### 8-3. `FILE_CATEGORY` 가 MySQL ENUM 인 경우

운영 DB에서 `TB_FILE_MASTER.FILE_CATEGORY` 가 **ENUM** 이면, 위 세 문자열이 목록에 **없으면** `ALTER TABLE` 로 ENUM 정의에 **추가**해야 INSERT 가 가능하다.

```sql
-- 예시: 기존 ENUM 에 항목을 덧붙이는 형태 (기존 값 목록은 실제 DB에 맞게 수정)
ALTER TABLE TB_FILE_MASTER
MODIFY COLUMN FILE_CATEGORY ENUM(
    /* ... 기존 값 ... */
    'Business_License',
    'Transportation_Business_License',
    'Insurance_Policy',
    'BUS_PHOTO'
) NOT NULL COMMENT '파일 분류';
```

`busTaams_server` 의 `ensureTbFileMasterTable` 는 **`FILE_CATEGORY VARCHAR(50)`** 로 테이블을 만들거나 짧은 ENUM 을 **VARCHAR 로 확장**하므로, **신규 환경**에서는 별도 ENUM 수정 없이 위 문자열을 그대로 넣을 수 있다. **이미 ENUM 으로 쓰는 DB**는 위와 같이 마이그레이션한다.

### 8-4. GCS 객체 명명 규칙 (요약)

- 물리 파일명(객체 키의 파일 부분)은 **`FILE_UUID`** 를 사용한다 (예: `{uuid}.pdf`).
- 원래 파일명은 **`ORG_FILE_NM`** 과 **`FILE_EXT`** 로만 DB에 보관하고, 버킷 경로에는 반드시 UUID 기반 이름을 쓴다.

### 8-4a. Google Cloud Storage 버킷·폴더 (`busTaams_server` 업로드 기준)

버킷 이름은 환경 변수 **`GCS_BUCKET_NAME`** (기본값 예: `bustaams-secure-data`) 이다. 아래는 **객체 키(폴더/파일 경로)** 만 정리한 것이며, 실제 URL은 `gs://{버킷}/{GCS_PATH}` 형태이다.

| 구분 | GCS 경로 패턴 (접두 폴더) | `FILE_CATEGORY` / 비고 |
|------|---------------------------|-------------------------|
| 회원가입 전자서명 | `signatures/{YYYYMM}/{FILE_UUID}.png` | 약관 서명 등 (`TB_FILE_MASTER` 연동) |
| 기사 프로필 사진 | `profiles/driver/{YYYY-MM}/{FILE_UUID}.png` | `DRIVER_PROFILE_PHOTO` |
| **운송종사자 자격증 사본(기사 프로필)** | `certificates/bus_licenses/{YYYY-MM}/{FILE_UUID}.{ext}` | **`Transport_Workers_Qualification_Information`** — 이력은 `TB_DRIVER_FILE_HIST` ([`기사정보등록_기사 화면.md`](./기사정보등록_기사%20화면.md) §2-1). 구 `BUS_QUAL_CERT` 는 DB에서 신규 코드로 이관됨. |
| (구) 운수자격 표기 | 동일 경로 | 레거시 문서의 `BUS_QUAL_CERT` 표기는 위 코드로 통일하는 것을 권장 |
| (구) 프로필/면허 임시 경로 | `certificates/{...}` 일부 레거시 | 구 `POST /api/driver/profile` 경로 |
| **차량 서류** (사업자·허가·보험) | **`vehicle_docs/{YYYY-MM}/{FILE_UUID}.{ext}`** | `Business_License` 등 + `TB_DRIVER_BUS_FILE_HIST` |
| **차량 사진** | **`vehicle_photos/{YYYY-MM}/{FILE_UUID}.{ext}`** | **`BUS_PHOTO`** + `TB_DRIVER_BUS_FILE_HIST` + `VEHICLE_PHOTOS_JSON` |

월 단위 `{YYYY-MM}` 또는 `{YYYYMM}` 은 업로드 시점 기준으로 나누어 객체를 분산한다. **차량 정보 등록** 화면에서만 쓰는 경로는 **`vehicle_docs/`** 와 **`vehicle_photos/`** 이다.

### 8-5. 서류 이력·버전 관리 (최소 요구사항)

사업자 등록증·운송사업 허가증·보험증권은 **갱신·변경·재발급**이 수시로 발생하므로, **단순히 `TB_DRIVER_BUS`의 FK만 덮어쓰기**만으로는 **누가 언제 어떤 서류를 제출했는지**를 추적할 수 없다. 다음을 **최소 이력**으로 만족시킨다.

| 원칙 | 설명 |
|------|------|
| **재등록 = 신규 버전** | 동일 슬롯(동일 `FILE_CATEGORY`)에 파일을 다시 올릴 때마다 **새 `FILE_UUID`** 를 발급하고, **GCS 업로드 → `TB_FILE_MASTER` INSERT** 를 수행한다. |
| **현재 유효 문서** | `TB_DRIVER_BUS` 의 `BIZ_REG_FILE_UUID` / `TRANS_LIC_FILE_UUID` / `INS_CERT_FILE_UUID` 는 **항상 “현재 유효한 최종 제출분”** 한 건만 가리킨다. |
| **과거 제출분 보존** | 이전 `FILE_UUID` 행은 `TB_FILE_MASTER` 에 **삭제하지 않고 유지**하되, “같은 차량·같은 서류 종류의 연속 버전”을 식별하려면 아래 **이력 연결**이 필요하다. |

**이력 연결(택1·최소):**

1. **이력 전용 테이블** 예: `TB_DRIVER_BUS_FILE_HIST`  
   - `BUS_UUID`, `FILE_CATEGORY`(`Business_License` 등), `FILE_UUID`, `REG_DT` (필수)  
   - 재업로드 시마다 **한 행 INSERT** 후, `TB_DRIVER_BUS` 의 해당 FK를 최신 `FILE_UUID` 로 **UPDATE**.  
   - 조회 시 `BUS_UUID` + `FILE_CATEGORY` 로 정렬하면 **버전 순 이력**을 얻는다.  
   - **CREATE TABLE 스크립트:** `busTaams_server/sql/tb_driver_bus_file_hist.sql` (실행 전 `TB_DRIVER_BUS`, `TB_FILE_MASTER` 가 있어야 함)

2. **`TB_FILE_MASTER` 확장**  
   - `BUS_UUID`, `FILE_CATEGORY`, (선택) `PREV_FILE_UUID` 또는 `VERSION_SEQ` 등을 두어 **동일 슬롯의 이전·이후**를 연결한다.

**재업로드 처리 순서(권장):** 새 `FILE_UUID` 발급 → GCS 저장 → `TB_FILE_MASTER` INSERT → **이력 테이블 INSERT**(위 1번) → `TB_DRIVER_BUS` 해당 FK 갱신.  
이전 GCS 객체는 **감사·분쟁 대비 보관**을 기본으로 하고, 스토리지 비용·개인정보 정책에 따라 **보존 기간 후 삭제** 여부를 별도 정한다.

**차량 사진(`BUS_PHOTO`):** 서류 3종과 동일하게 **매 업로드마다** `TB_FILE_MASTER` + **`TB_DRIVER_BUS_FILE_HIST`** (`FILE_CATEGORY` = `BUS_PHOTO`) 에 쌓고, **현재 슬롯에 노출할 8장**은 `TB_DRIVER_BUS.VEHICLE_PHOTOS_JSON` 에 `FILE_UUID` 문자열 배열로 저장한다. 사진을 교체·재등록하면 **새 `FILE_UUID`** 가 이력에 추가되고, JSON 은 **최신 등록분**으로 갱신된다. 재등록 API는 **`PATCH /api/driver/bus/photos`** (본문 `vehiclePhotos` 배열, 최대 8개).

### 8-6. 원본 조회·다운로드 권한

서류 원본은 **민감·증빙 자료**이므로 노출 주체를 다음과 같이 제한한다.

| 주체 | 조회·다운로드(원본 파일) | 비고 |
|------|---------------------------|------|
| **해당 차량 소유 기사** (`TB_DRIVER_BUS.USER_UUID` 와 동일한 로그인 사용자) | **허용** — 등록된 **최종(현재 FK가 가리키는) 문서**를 **원본 그대로** 미리보기·다운로드할 수 있다. 필요 시 **이력 API**로 과거 제출분 조회는 **정책에 따라** 허용(동일 기사 한정). |
| **다른 버스 기사** | **불가** — 조회·출력·다운로드 API 미제공 또는 **403**. |
| **영업 회원** | **불가** (동일). |
| **여행자(일반 이용자)** | **불가** (동일). |

**구현 시:** 파일 스트리밍·서명 URL 발급 API는 **인증된 사용자의 `USER_UUID`** 와 **요청 대상 `FILE_UUID`(또는 `BUS_UUID`)** 를 서버에서 검증하여, **차량 소유 기사와 일치할 때만** GCS에서 읽어 응답한다. 프론트는 **다운로드 링크를 타 역할 화면에 노출하지 않는다.**

### 8-7. 이력 테이블 DDL 경로

| 항목 | 경로 |
|------|------|
| **`TB_DRIVER_BUS_FILE_HIST` 생성 스크립트** | `busTaams_server/sql/tb_driver_bus_file_hist.sql` |

운영 DB에 적용할 때는 위 파일을 열어 **선행 테이블**(`TB_DRIVER_BUS`, `TB_FILE_MASTER`) 존재 여부·FK 호환을 확인한 뒤 실행한다.

## 9. 차량 사진 (05. 차량 사진)

**04. 검사 및 서류** 바로 아래에 **05. 차량 사진** 구역을 둔다. 외관·실내 등 차량을 잘 보이는 **이미지**를 **최대 8장**까지 등록한다.

### 9-1. UI (`BusInformationSetup.jsx`)

- **제목:** `05. 차량 사진`
- **슬롯:** 고정 **8개** (`grid` — 모바일 2열·넓은 화면 4열). 슬롯마다 `사진 1` … `사진 8` 라벨 + `add_a_photo` 아이콘.
- **기존 등록분:** `GET /api/driver/bus` 의 `vehiclePhotoFileUuids` 순서로 슬롯을 채우고, 썸네일은 **`GET /api/driver/bus-documents/file`** URL(`object-contain`, 슬롯 높이에 맞춤).
- **선택 시:** 썸네일 미리보기 + 파일명, **제거** 버튼으로 해당 슬롯만 비움.
- **형식:** JPG / PNG / WebP, **슬롯당 최대 5MB** (서류와 동일 상한).
- **제출:** 신규 등록은 빈 슬롯을 제외하고 `POST /api/driver/bus` 에 `vehiclePhotos` 배열. 수정 모드는 **`PATCH /api/driver/bus/photos`** 에 기존 `{ fileUuid }` 와 신규 `{ base64, fileName }` 를 순서대로 혼합.
- **상단 표시:** `등록 N / 8` (현재 채워진 슬롯 수).

### 9-2. 저장·GCS·이력

| 항목 | 내용 |
|------|------|
| `FILE_CATEGORY` | **`BUS_PHOTO`** (ENUM/`VARCHAR` 에 동일 문자열 등록) |
| `TB_FILE_MASTER` | GCS 메타데이터, 경로 예: **`vehicle_photos/{YYYY-MM}/{FILE_UUID}.jpg`** (§8-4a) |
| **`TB_DRIVER_BUS_FILE_HIST`** | 사진 **매 장마다** INSERT — 서류와 같은 이력 테이블, `FILE_CATEGORY` = `BUS_PHOTO` |
| `TB_DRIVER_BUS` | **`VEHICLE_PHOTOS_JSON`** — 현재 화면에 올린 **최대 8장**의 `FILE_UUID` 문자열 배열(순서 유지). 재등록 시마다 새 행이 이력에 쌓이고 JSON 은 최신 세트로 갱신된다. |
| 스키마 보장 | `ensureTbDriverBusVehiclePhotosColumn` — `VEHICLE_PHOTOS_JSON` 컬럼 없으면 `ALTER` |

### 9-3. API 요약

- **`POST /api/driver/bus`** — 본문 **`vehiclePhotos`**: `[ { base64, fileName }, ... ]` 최대 8개. 성공 시 **`vehiclePhotoFileUuids`** (선택).
- **`PATCH /api/driver/bus/photos`** — 등록된 차량(`busUuid`)의 사진만 교체·추가할 때. 이력 누적 + `VEHICLE_PHOTOS_JSON` 갱신 (**§5-6**).
- **이력 조회:** **§5-7** `GET /api/driver/bus-documents/history` — `BUS_PHOTO` 행 포함.
- **원본 보기:** **§5-8** `GET /api/driver/bus-documents/file` — 이력에 있는 파일이면 스트리밍(구 `Vehicle_Photo` + JSON 만 있는 레거시는 §5-8 호환 분기).
