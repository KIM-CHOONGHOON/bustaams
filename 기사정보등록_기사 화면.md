# 기사정보등록 (Driver Profile Setup) 화면 명세

본 문서는 `busTaams` 프로젝트의 기사 정보 등록 화면에 대한 UI·데이터·스토리지·외부 연동을 정의합니다. **GCS 버킷·폴더 구조**는 [`ARCHITECTURE.md`](./ARCHITECTURE.md)의 **「파일 및 스토리지 아키텍처」**를 기준으로 합니다. **테이블·컬럼 ID·타입·길이의 정본**은 [`BusTaams 테이블.md`](./BusTaams%20테이블.md) 를 따릅니다.

---

## 0. 회원·기사 식별자 (REST / DB 정합)

| 항목 | 설명 |
| :--- | :--- |
| **권장 JSON·쿼리 키** | **`userId`** — `TB_USER` 의 **`USER_ID`(로그인 ID)** 또는 **`CUST_ID`(10자리 내부 식별자)** 와 **동일한 문자열**을 씁니다(세션/클라이언트 DTO, `buildPostLoginUserDto` 등). |
| **하위 호환** | `userUuid` 쿼리/본문 필드는 일부 API에서 **동의어**로만 수용합니다. 신규 연동은 **`userId`만** 보내는 것을 권장합니다. |
| **`TB_DRIVER_INFO` PK** | **`USER_ID` VARCHAR(256)** — `busTaams_server/server.js` 의 `ensureDriverInfoTable` 이 생성·사용하는 정본(레거시 `USER_UUID` BINARY PK가 아님). |
| **타 열과의 명칭** | 설계서상 `TB_AUCTION_REQ`.`TRAVELER_ID`, `TB_BUS_RESERVATION`.`DRIVER_ID`, `TB_TRIP_REVIEW`.`WRITER_ID`·`DRIVER_ID` 등은 **UUID 컬럼이 아닌 `VARCHAR`** 키로 통일한 경우가 있으며, API 필드명은 **위 표의 `userId` / 도메인별 `travelerId` 등**으로 매핑합니다(상세는 `BusTaams 테이블.md`). |

## 1. 화면 개요

- **목적:** 기사 전문성 검증을 위한 면허·자격·프로필 등 추가 정보 수집
- **접근:** 드라이버 대시보드 > 바로가기 > **「기사 정보」**
- **구현:** `busTaams_web/src/components/DriverProfileSetup/DriverProfileSetup.jsx`
- **디자인:** The Radiant Traveler 가이드라인

---

## 2. 업로드 파일 — Google Cloud Storage (GCS)

버킷 기본값: `bustaams-secure-data` (리전 `asia-northeast3`). 업로드는 **백엔드**에서 수행하며, DB에는 `TB_FILE_MASTER`에 메타데이터만 저장합니다.

| 구분 | GCS 객체 prefix (예시) | 비고 |
| :--- | :--- | :--- |
| **프로필 사진** | `profiles/driver/{YYYY-MM}/tmp_profile_{timestamp}.png` 등 | 업로드 후 `TB_FILE_MASTER`에 `FILE_ID`(varchar(20) 0패딩)·GCS 경로 저장. `FILE_CATEGORY`: `DRIVER_PROFILE_PHOTO` |
| **운송종사자 자격증 사본** | `certificates/bus_licenses/{YYYY-MM}/qual_{timestamp}.{png\|pdf}` | [`ARCHITECTURE.md`](./ARCHITECTURE.md) **2-2 [기사 서류]** 계열. `FILE_CATEGORY`: `BUS_QUAL_CERT`. DB 식별자는 **`FILE_ID`** (UUID 아님). |

- **접근 권한:** Private — 운영 시 Signed URL 또는 백엔드 경유 열람.
- **금지:** DB BLOB에 Base64 원문 적재 (아키텍처 정책과 동일).

---

## 3. 주민등록번호 입력 규칙

- **수집 형식:** **앞 6자리 + 하이픈 + 뒤 1자리**만 허용 (예: `900101-1`).
- **의미:** 생년월일(앞 6자리)과 성별·내·외국인 등을 나타내는 **뒷자리 첫 번째 숫자 1자리**만 입력받아 불필요한 식별자 수집을 줄입니다.
- **저장:** 서버에서 `AES-256-GCM` 양방향 암호화 후 `TB_DRIVER_INFO.RRN_ENC` 등에 저장합니다.
- **검증:** API `POST /api/driver/profile-setup` 에서 정규식 `^\d{6}-\d{1}$` 로 검사합니다.

## 4. 휴대전화 인증 — Firebase Phone Authentication

기사 정보 화면과 **회원가입** 화면 모두 **Firebase Authentication(Phone)** 으로 SMS를 보내고, 인증 완료 후 발급되는 **Firebase ID 토큰**을 백엔드(`firebase-admin`)가 검증하는 흐름을 맞춥니다. 서버 쪽 검증 함수는 **`verifyFirebasePhoneIdTokenIfRequired`** 하나로 **`POST /api/auth/register`** 와 **`POST /api/driver/profile-setup`** 에 공통 적용됩니다.

### 4-1. 회원가입 화면 (`SignupPage.jsx`)

| 단계 | 내용 |
| :--- | :--- |
| **초기화** | `App.jsx` 에서 `initializeApp(firebaseConfig)` 후 `window.auth = getAuth(app)` , `window.firebaseAuth = { RecaptchaVerifier, signInWithPhoneNumber }` 를 노출합니다. |
| **reCAPTCHA** | `#recaptcha-signup-wrapper` 컨테이너에 대해 `new RecaptchaVerifier(window.auth, 'recaptcha-signup-wrapper', { size: 'invisible' })` 를 생성합니다. |
| **SMS 발송** | `signInWithPhoneNumber(window.auth, E.164번호, recaptchaVerifier)` — 국내 번호는 `+82` 로 변환(예: `010…` → `+8210…`). |
| **코드 확인** | `confirmationResult.confirm(인증번호)` → `user.getIdToken()` 으로 ID 토큰 확보 → **`signOut(window.auth)`** 로 웹 세션의 Firebase 로그인 상태를 정리(회원가입은 자체 DB 로그인을 쓰므로). |
| **가입 요청** | `POST /api/auth/register` 본문에 **`firebaseIdToken`** 필드로 위 토큰을 실어 보냅니다. |
| **로컬 생략** | `.env`에 `VITE_SKIP_FIREBASE_PHONE=true` 이면 SMS 호출을 건너뜁니다. 이 경우 **서버에 `FIREBASE_SERVICE_ACCOUNT_PATH` 가 없을 때만** 가입이 허용됩니다(Admin이 있으면 토큰 검증 실패). |

> 이전에는 **테스트 모드**로 `signInWithPhoneNumber` 호출이 주석 처리되어 있었고, SMS 없이 `isPhoneVerified` 만 `true` 로 두는 방식이었습니다. 현재는 위와 같이 **실제 SMS 경로를 기본**으로 두고, 필요 시에만 `VITE_SKIP_FIREBASE_PHONE` 으로 생략합니다.

### 4-2. 운전 기사 기본 정보 (`DriverProfileSetup.jsx` + `firebasePhoneVerify.js`)

- **발송:** `signInWithPhoneNumber` 로 SMS 발송. reCAPTCHA는 **invisible** 컨테이너(`#driver-phone-recaptcha`)를 사용합니다.
- **인증 앱:** 로그인용 기본 Firebase 앱과 충돌하지 않도록 `firebasePhoneVerify.js` 에서 **별도 앱 이름**(`phoneVerifyDriver`)으로 `Auth` 인스턴스를 분리합니다.
- **확인:** 6자리 입력 후 **「인증 확인」** → `confirmationResult.confirm(code)` → ID 토큰 → `signOut(phoneAuth)` 로 해당 전용 Auth 세션 정리.
- **서버:** 제출 시 **`phoneIdToken`** 필드로 전송. `POST /api/driver/profile-setup` 에서 회원가입과 **동일한** `verifyFirebasePhoneIdTokenIfRequired` 로 검증합니다.
- **Admin 미설정:** 토큰 없이도 저장 가능(로컬 개발). **Admin 설정 시**에는 토큰 필수.

### 4-3. 백엔드 공통 (`busTaams_server/server.js`)

- **`firebaseAdminConfigured()`** — `FIREBASE_SERVICE_ACCOUNT_PATH` 파일 존재 여부.
- **`verifyFirebasePhoneIdTokenIfRequired(idToken)`** — Admin이 설정된 경우에만 `admin.auth().verifyIdToken(idToken)` 실행; 미설정이면 검증 생략.
- **적용 API:** `POST /api/auth/register` (`firebaseIdToken`), `POST /api/driver/profile-setup` (`phoneIdToken`).

- **참고:** [`ARCHITECTURE.md`](./ARCHITECTURE.md) **외부 서비스 연동 — Firebase Phone Auth** 절.

## 5. 운전면허 정보 — 진위 확인 (백엔드 연동)

**목적:** 면허번호·성명·생년월일·**암호일련번호** 등을 대조하여 유효 면허 여부 확인.

| 항목 | 내용 |
| :--- | :--- |
| **데이터 출처** | 도로교통공단·경찰청 등(서비스별 상이) |
| **공공 오픈 API** | **「운전면허정보 자동검증시스템」** 은 자동차대여사업자 등 **승인된 기관용 B2B** 로 공개되는 경우가 많아, 포털 JSON 한 종류로 고정하기 어렵습니다. |
| **본 프로젝트 구현** | `busTaams_server/driverVerification.js` 의 **`verifyKoroadDriverLicense`** — 기관에서 받은 **전용 검증 URL**(`KOROAD_LICENSE_VERIFY_URL`)로 **JSON POST** 합니다. 응답에 `success`·`valid`·`resultCode === '00'` 등 중 하나가 성공으로 인정되도록 처리합니다(실제 스키마는 연동 기관 명세에 맞게 조정). |
| **활성화** | `KOROAD_LICENSE_VERIFY_ENABLED=true` 일 때만 저장 전에 호출합니다. **기본값은 비활성**이라 기존 배포는 그대로 동작합니다. |
| **입력 필드** | 면허번호(`licenseNo`), 암호일련번호(`licenseSerialNo`), 성명(`driverName`), 주민번호(앞6+뒤1)로부터 유도한 **생년월일 8자리** |

| **조회** | 화면 마운트 시 `GET /api/driver/profile-setup?userId=` 호출 (`userUuid=` 는 동의어) |
| **`exists: false`** | `TB_DRIVER_INFO`에 행이 없음 — `TB_USER` 기준 **성명·휴대전화**만 채움. 면허·자격 필드는 신규 입력(잠금 없음) |
| **`exists: true`** | 주민번호 앞6·뒤1, 면허(종류·번호·암호일련·발급·만료), 버스운전 자격번호, 자기소개 등으로 폼을 채움. **운전면허 정보**·**운송종사자 자격**(자격번호·자격증 업로드 영역)은 기본 **읽기 전용**이며, 각 섹션의 **「수정」**으로 잠금 해제 후 편집 가능 |
| **프로필·자격 파일** | `hasProfilePhoto` / `hasQualCertFile`이 true이면, 새 파일을 선택하지 않아도 기존 서버 파일이 유지됨(안내 문구 표시) |

### 7-4. 진위 검증 생략 규칙 (백엔드)

저장(`POST`) 직전, 서버는 `TB_DRIVER_INFO`에서 조회한 **기존 행**과 이번 요청 본문을 비교합니다(`busTaams_server/driverVerification.js`).

| 구분 | 비교 필드 | 동일할 때 |
| :--- | :--- | :--- |
| **운전면허 진위 (KOROAD)** | `LICENSE_TYPE`, `LICENSE_NO`, `LICENSE_SERIAL_NO`, `LICENSE_ISSUE_DT`, `LICENSE_EXPIRY_DT` | `KOROAD_LICENSE_VERIFY_ENABLED=true`여도 **면허 진위 API를 호출하지 않음** |
| **운수종사자 자격 (TS)** | `QUAL_CERT_NO`(버스운전 자격번호) | `TS_QUAL_VERIFY_ENABLED=true`여도 **자격 진위 API를 호출하지 않음** |

- **최초 등록:** 기존 행이 없으면 위 비교가 성립하지 않으므로, 플래그가 켜져 있으면 진위 검증을 수행합니다.
- **수정:** 화면에서 「수정」을 눌렀다가 **값을 바꾸지 않고** 제출해도, 요청 값이 DB와 같으면 동일하게 **진위 API를 생략**합니다. 면허 또는 자격번호 중 **하나라도** 바뀌면 해당 항목만 외부 검증 대상이 됩니다.

### 7-5. 하단 제출 버튼 · 성공 모달

| 항목 | 내용 |
| :--- | :--- |
| **버튼 문구** | `GET` 결과 `exists: false`(최초 등록)일 때 **「등록」**, `exists: true`(이미 `TB_DRIVER_INFO`에 행 있음)일 때 **「수정」**. 제출 중에는 **「처리 중...」** |
| **성공 후 동작** | 저장 성공 시 **대시보드 등으로 자동 이동하지 않음**. 같은 화면에 머무르며 **모달(팝업)** 으로만 안내합니다. |
| **최초 등록 성공** | 모달 메시지: **「버스 기사 기본 정보가 등록되었습니다.」** — 확인 시 모달만 닫힘. 이후 동일 세션에서도 `profileExistsOnServer` 상태를 갱신하여 주요 버튼이 **「수정」**으로 바뀌고, 면허·자격 블록은 다시 잠금 처리됩니다. |
| **수정 성공** | 모달 메시지: **「버스 기사 기본 정보가 수정 되었습니다.」** |
| **취소** | **「취소」**는 기존과 같이 상위에서 넘긴 `onBack`으로 대시보드 등 이전 화면으로 돌아갑니다(제출 버튼과 무관). |

## 8. 백엔드 API — `GET /api/driver/profile-setup`

- **목적:** 등록된 기사 정보를 불러와 폼에 채움.
- **쿼리:** `userId` (필수, 권장) — `userUuid` (선택, 동의어; 동일한 문자열을 받음)
- **동작:** `TB_USER`에서 성명·휴대전화(복호화), `TB_DRIVER_INFO`(`USER_ID` = 위 식별 문자열)에서 면허·자격·자기소개·`PROFILE_FILE_ID` / `QUAL_CERT_FILE_ID` 존재 여부. 테이블/행이 없으면 `exists: false`와 함께 사용자 정보만 반환할 수 있음.

**응답 필드 (요약):**

| 필드 | 설명 |
| :--- | :--- |
| `exists` | `TB_DRIVER_INFO`에 해당 사용자 행이 있으면 `true` |

## 9. 백엔드 API — `POST /api/driver/profile-setup`

- **기능:** 주민번호 형식 검증, **Firebase ID 토큰 검증**(`verifyFirebasePhoneIdTokenIfRequired`, 회원가입과 동일), **선택적 면허·운수자격 진위**(`runDriverVerificationsForProfileSetup` → `driverVerification.js`), GCS 업로드, `TB_FILE_MASTER`·`TB_DRIVER_INFO` Upsert, 이어서 **`TB_USER.CUST_ID`**가 있으면 **`TB_DRIVER_DOCS`**에 `LICENSE` / `QUALIFICATION` 행을 UPSERT(본사 승인·서류 관리용, [`BusTaams 테이블.md`](./BusTaams%20테이블.md) §`TB_DRIVER_DOCS`)

## 11. 데이터베이스

### 11-1. 논리 스키마 (문서)

- **폼/조회 1차 저장:** `TB_USER`, **`TB_DRIVER_INFO`**(기사 본인·면허·자격·프로필 `FILE_ID` 등). 상세는 [`BusTaams 테이블.md`](./BusTaams%20테이블.md) 및 `server.js`의 `ensureDriverInfoTable` 와 **컬럼 ID·길이**를 맞출 것.
- **본사 심의·서류 승인:** [`BusTaams 테이블.md`](./BusTaams%20테이블.md) **`TB_DRIVER_DOCS`** — PK **`(CUST_ID, DOC_TYPE)`**. `CUST_ID` = `TB_USER.CUST_ID`(varchar(10)). `DOC_TYPE`은 `LICENSE`·`QUALIFICATION` 등(공통코드 `DOC_TYPE`과 동일). 면허/자격 **문서번호**는 `DOC_NO_ENC`(암호화), 자격 **첨부**는 `FILE_ID` → `TB_FILE_MASTER`.
- **프로필 사진**은 `TB_DRIVER_INFO.PROFILE_FILE_ID`에만 두고, `TB_DRIVER_DOCS` ENUM에는 **프로필 전용 `DOC_TYPE`이 없으므로** 이 테이블에 프로필 행을 만들지 않습니다.
- **면허 암호일련번호(진위 API용):** `TB_DRIVER_INFO.LICENSE_SERIAL_NO` (문서 `TB_DRIVER_DOCS`에는 별도 컬럼 없음).

### 11-2. 물리 테이블 자동 생성 (서버)

로컬·신규 DB처럼 테이블이 아직 없어도 **`POST /api/driver/profile-setup`**(및 서버 기동 시 `startServer` 마이그레이션)에서 `busTaams_server/server.js` 가 다음을 수행합니다.

| 순서 | 대상 | 동작 |
| :--- | :--- | :--- |
| 1 | `TB_USER` | 없으면 `CREATE` (CUST_ID·USER_ID 정본, `ensureTbUserTable`) |
| 2 | `TB_DRIVER_INFO` | PK **`USER_ID` VARCHAR(256)**, `RRN_ENC`, 면허/자격 필드, **`QUAL_CERT_FILE_ID` / `PROFILE_FILE_ID` VARCHAR(20)** → `TB_FILE_MASTER.FILE_ID` FK, `BIO_TEXT`, `UPDATE_DT` 등 (`ensureDriverInfoTable`) |
| 3 | `TB_FILE_MASTER` | PK **`FILE_ID` VARCHAR(20)**(0패딩), GCS 메타 (`ensureTbFileMasterTable`) |
| 4 | `TB_DRIVER_INFO` 마이그레이션 | `ensureDriverInfoLicenseSerialColumn`, `ensureDriverInfoQualVerifyColumns`, `ensureDriverInfoFileIdColumns` — **없는 열만** `ALTER` |
| 5 | `TB_DRIVER_DOCS` | `BusTaams 테이블.md` 정본에 맞춘 `CREATE IF NOT EXISTS` (`ensureTbDriverDocsTable`) |

- **`POST` 트랜잭션:** `TB_DRIVER_INFO` UPSert 후, `resolveCustIdByUserKey(USER_ID)`로 **`CUST_ID`**를 구할 수 있으면, 같은 트랜잭션에서 **`TB_DRIVER_DOCS`에 `LICENSE`·`QUALIFICATION` UPSERT** — 면허는 `LICENSE_TYPE_CD`·`ISSUE_DT`·`EXP_DT`·`DOC_NO_ENC(면허번호 암호화)` 등, 자격은 `FILE_ID`·`DOC_NO_ENC(자격번호 암호화)` 등(구현: `syncTbDriverDocsFromProfileConnection`).
- **CUST_ID 미해결** 시(로그인 키만 있고 `TB_USER`에 매칭 없음)에는 `TB_DRIVER_DOCS` 동기화를 **건너뜁니다** (FK 제약·데이터 정합).
- 운영 DB는 DBA가 [`BusTaams 테이블.md`](./BusTaams%20테이블.md)와 `SHOW CREATE TABLE` diff 후 이관하는 것이 권장됩니다.

## 13. 버스운전 자격번호 검증 상태 관리

### 13-1. 설계 목적

자격증 사본 파일 업로드는 **버스운전 자격번호(`QUAL_CERT_NO`) 검증이 정상 완료된 이후에만 허용**합니다.
기사는 기본 정보를 먼저 등록(사본 없이)한 뒤, 나중에 모달을 다시 열어 자격증 사본을 업로드할 수 있습니다.
이를 위해 검증 완료 여부를 DB에 영속적으로 관리합니다.

### 13-2. 관리 테이블: `TB_DRIVER_INFO`

기존 `TB_DRIVER_INFO`에 **2개 컬럼을 추가**하여 검증 상태를 관리합니다. 별도 이력 테이블 없이 최신 상태 1건만 유지합니다.

| 컬럼 | 타입 | 기본값 | 설명 |
| :--- | :--- | :--- | :--- |
| `QUAL_CERT_VERIFY_STATUS` | `VARCHAR(20) NOT NULL` | `'UNVERIFIED'` | 자격번호 검증 상태 (아래 값 참조) |
| `QUAL_CERT_VERIFY_DT` | `DATETIME NULL` | `NULL` | 검증 완료 일시 (`VERIFIED` 전환 시 기록) |

**상태 값 정의:**

| 값 | 의미 | 사본 업로드 허용 |
| :--- | :--- | :---: |
| `UNVERIFIED` | 아직 검증 안 됨 (기본값) | ❌ |
| `VERIFIED` | TS 공공 API 검증 성공 | ✅ |
| `SKIPPED` | `TS_QUAL_VERIFY_ENABLED=false` — API 비활성(개발·로컬 환경) | ✅ |

### 13-3. 상태 전환 규칙 (`POST /api/driver/profile-setup`)

| 조건 | 새 상태 |
| :--- | :--- |
| `TS_QUAL_VERIFY_ENABLED=false` | `SKIPPED` |
| TS API 활성화 + 자격번호 **변경 없음** (DB와 동일) | 기존 DB 상태 유지 |
| TS API 활성화 + 자격번호 **신규·변경** + API 성공 | `VERIFIED` + `QUAL_CERT_VERIFY_DT = NOW()` |
| TS API 활성화 + API **실패** | POST 400 에러 반환 — DB 저장 불가 (상태 변경 없음) |

### 13-4. 프론트엔드 동작 (`DriverProfileSetup.jsx`)

- `GET /api/driver/profile-setup` 응답의 `qualCertVerifyStatus` 값을 `qualCertVerifyStatus` state에 세팅합니다.
- **자격증 사본 업로드 영역**은 「차량정보등록」 모달과 동일하게, 다음 경우에 활성화됩니다:
  - **신규 등록** — 프로필이 서버에 없는 경우 (`!profileExistsOnServer`)
  - **수정 모드** — 기존 프로필이 있고 「수정」 버튼을 눌러 잠금 해제된 경우 (`!qualFieldsLocked`)
- 반대로 기존 프로필이 있고 잠금 상태(`qualFieldsLocked === true`)이면 업로드 영역이 비활성화됩니다. 「수정」 버튼을 먼저 클릭하면 활성화됩니다.
- `qualCertVerifyStatus`는 **업로드 차단 조건이 아닙니다**. VERIFIED / SKIPPED / UNVERIFIED 여부와 관계없이 위 잠금 조건만 적용됩니다. 검증 상태는 정보성 배지로만 표시됩니다.
- `POST` 성공 응답의 `qualCertVerifyStatus`, `qualCertFileId`로 state를 실시간 갱신합니다.

### 13-5. 마이그레이션 (기존 DB)

서버 시작 시 `ensureDriverInfoQualVerifyColumns(connection)`이 자동 호출되어, 컬럼이 없으면 `ALTER TABLE`로 추가합니다. 기존 행의 `QUAL_CERT_VERIFY_STATUS`는 DEFAULT `'UNVERIFIED'`로 채워집니다.

> 운영 DB에서는 DBA가 사전에 아래 DDL을 실행하는 것을 권장합니다.

### 13-6. 최종 `TB_DRIVER_INFO` CREATE TABLE 쿼리 (서버 `ensureDriverInfoTable` + 마이그레이션 열)

아래는 **로컬/신규 DB**에 서버가 `CREATE IF NOT EXISTS` 로 만드는 형태이며, `QUAL_CERT_VERIFY_*` / `LICENSE_SERIAL_NO` / 파일 UUID 열은 상황에 따라 `ensureDriverInfo*` 로 `ALTER` 됩니다. PK 는 **`USER_ID` VARCHAR(256)** = `TB_USER` 의 로그인·내부 키와 동일한 문자열입니다.

```sql
-- 서버 `ensureDriverInfoTable` + 파일/검증 컬럼 마이그레이션과 합쳐 본 정본 (FILE_ID = varchar(20) 0패딩)
CREATE TABLE IF NOT EXISTS TB_DRIVER_INFO (
    USER_ID                 VARCHAR(256)  NOT NULL
                                            COMMENT 'TB_USER.USER_ID 또는 동일정책 키',
    RRN_ENC                 VARCHAR(512)  NOT NULL,
    LICENSE_TYPE            VARCHAR(50)   NULL,
    LICENSE_NO              VARCHAR(100)  NULL,
    LICENSE_SERIAL_NO       VARCHAR(100)  NULL
                                            COMMENT '면허 암호일련번호(진위검증용)',
    LICENSE_ISSUE_DT        DATE          NULL,
    LICENSE_EXPIRY_DT       DATE          NULL,
    QUAL_CERT_NO            VARCHAR(100)  NULL
                                            COMMENT '버스운전 자격번호',
    QUAL_CERT_VERIFY_STATUS VARCHAR(20)   NOT NULL DEFAULT 'UNVERIFIED'
                                            COMMENT 'UNVERIFIED/VERIFIED/SKIPPED',
    QUAL_CERT_VERIFY_DT     DATETIME      NULL
                                            COMMENT '자격 검증 완료 일시',
    QUAL_CERT_FILE_ID      VARCHAR(20)    NULL
                                            COMMENT 'TB_FILE_MASTER.FILE_ID',
    PROFILE_FILE_ID         VARCHAR(20)   NULL
                                            COMMENT 'TB_FILE_MASTER.FILE_ID',
    BIO_TEXT                TEXT          NULL,
    CREATE_DT               DATETIME      DEFAULT CURRENT_TIMESTAMP,
    UPDATE_DT               DATETIME      DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    PRIMARY KEY (USER_ID)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

**기존 테이블 마이그레이션 DDL (컬럼 추가):**

```sql
-- QUAL_CERT_VERIFY_STATUS 컬럼 추가 (이미 있으면 무시)
ALTER TABLE TB_DRIVER_INFO
    ADD COLUMN QUAL_CERT_VERIFY_STATUS VARCHAR(20) NOT NULL DEFAULT 'UNVERIFIED'
        COMMENT '버스운전 자격번호 검증 상태: UNVERIFIED/VERIFIED/SKIPPED';

-- QUAL_CERT_VERIFY_DT 컬럼 추가 (이미 있으면 무시)
ALTER TABLE TB_DRIVER_INFO
    ADD COLUMN QUAL_CERT_VERIFY_DT DATETIME NULL
        COMMENT '자격 검증 완료 일시 (VERIFIED 전환 시 기록)';
```

```
│  │    수정 모드(「수정」 클릭 후): 활성화                     │    │
│  │    조회 모드(잠금 상태): 비활성화 → 「수정」 버튼 안내     │    │
│  └──────────────────────────────────────────────────────────┘    │
│                                              [ 👁 문서 보기 ]     │  ← 오른쪽 정렬
└──────────────────────────────────────────────────────────────────┘
```

**파일 선택 영역 (업로드 버튼) 활성화 조건:**

| 상황 | 업로드 영역 |
|------|------------|
| 신규 등록 (`!profileExistsOnServer`) | **활성화** |
| 수정 모드 — 「수정」 버튼 클릭 후 (`!qualFieldsLocked`) | **활성화** |
| 조회 모드 — 기존 프로필 잠금 상태 (`qualFieldsLocked === true`) | **비활성화** → 「수정」 버튼 먼저 클릭 안내 메시지 표시 |

> `qualCertVerifyStatus`(VERIFIED / SKIPPED / UNVERIFIED)는 업로드 차단 조건이 아닙니다.  
> 검증 상태는 정보성 배지로만 표시됩니다.

**「문서 보기」 버튼 활성화 조건:**

| 상태 | 「문서 보기」 버튼 |
|------|-----------------|
| 서버에 파일 없음 (`hasQualCertFileOnServer === false`) | **비활성화** (opacity-60), 클릭 불가. hover tooltip: `서버에 등록된 파일이 없습니다` |
| 서버에 파일 있음 (`hasQualCertFileOnServer === true`) | **활성화** (secondary 색상). 클릭 시 CommonView 모달 오픈 |

### 14-3. CommonView 호출 파라미터

`DriverProfileSetup.jsx` 내부에서 CommonView를 직접 렌더링합니다.

```jsx
{showQualCertViewer && qualCertFileId && (
  <CommonView
    close={() => setShowQualCertViewer(false)}
    fileId={qualCertFileId}
    userId={commonViewUserId}
    docTitle="버스운전 자격증 사본"
    metaPath="/api/driver/qual-cert/meta"
    streamPath="/api/driver/qual-cert/file"
    downloadPath="/api/driver/qual-cert/download"
  />
)}
```

| prop | 값 | 설명 |
|------|-----|------|
| `fileId` | `qualCertFileId` (state) | GET 응답 `qualCertFileId` 또는 POST `qualCertFileId` (`TB_FILE_MASTER.FILE_ID`) |
| `userId` | `userId` → `custId` → `userUuid` → `uuid` 순으로 resolve 한 세션 키 | `TB_USER` 와 `TB_DRIVER_INFO.USER_ID` 가 일치하는 소유자 식별자. (선택) `userUuid` prop 은 **레거시 별칭**으로만 지원. |
| `fileUuid` | (미사용) | CommonView는 **`fileId` 우선**; 과거 `fileUuid` prop은 `TB_FILE_MASTER.FILE_ID` 문자열에 대한 레거시 이름이었음. |
| `docTitle` | `"버스운전 자격증 사본"` | 뷰어 상단 제목 |
| `metaPath` | `/api/driver/qual-cert/meta` | 자격증 전용 메타 API |
| `streamPath` | `/api/driver/qual-cert/file` | 자격증 전용 스트리밍 API |
| `downloadPath` | `/api/driver/qual-cert/download` | 자격증 전용 다운로드 API |

### 14-4. 관련 상태 변수

| 상태 | 타입 | 초기값 | 설명 |
|------|------|--------|------|
| `hasQualCertFileOnServer` | `boolean` | `false` | 서버에 파일 존재 여부 → 버튼 활성화 조건 |
| `qualCertFileId` | `string \| null` | `null` | CommonView `fileId` — `TB_FILE_MASTER.FILE_ID` |
| `showQualCertViewer` | `boolean` | `false` | 문서 보기 모달 표시 여부 |

**데이터 흐름:**
1. `GET /api/driver/profile-setup` 응답 → `qualCertFileId`, `hasQualCertFile` 상태 설정
2. `POST /api/driver/profile-setup` 성공 응답 → `qualCertFileId` 갱신 (자격증 파일이 새로 저장된 경우)

### 14-5. 서버 API — 자격증 사본 전용 엔드포인트

기존 차량 서류용 API(`/api/common-view/bus-document/*`, `/api/driver/bus-documents/file`)는  
`TB_BUS_DRIVER_VEHICLE`을 기준으로 소유 확인을 합니다.  
자격증 사본은 `TB_DRIVER_INFO.QUAL_CERT_FILE_ID`에 저장되므로 **별도 전용 엔드포인트**를 사용합니다.

| 메서드·경로 | 용도 | 소유 확인 테이블 |
|------------|------|----------------|
| `GET /api/driver/qual-cert/meta` | CommonView doc mode 메타 조회 | `TB_DRIVER_INFO.QUAL_CERT_FILE_ID` = `FILE_ID` |
| `GET /api/driver/qual-cert/file` | 파일 스트리밍 (인라인 미리보기) | 동일 |
| `GET /api/driver/qual-cert/download` | 파일 다운로드 (attachment) | 동일 |

**공통 쿼리 파라미터:** `userId`(권장), **`fileId`**(권장) — `fileUuid` 는 **동의어(레거시)**  
**소유 확인 헬퍼:** `canAccessQualCertFile(connection, userIdStr, fileIdStr)` — `TB_DRIVER_INFO.USER_ID` + `QUAL_CERT_FILE_ID` 기준

### 14-6. CommonView 카테고리 레이블

`CommonView.jsx`의 `COMMON_VIEW_CATEGORY_LABEL` 맵에 아래 항목이 추가되어 있습니다.

```js
BUS_QUAL_CERT: '버스운전 자격증 사본',
```

`TB_FILE_MASTER.FILE_CATEGORY`에 `'BUS_QUAL_CERT'`로 저장된 파일은 CommonView 사이드바·제목에  
**「버스운전 자격증 사본」** 으로 표시됩니다.

### 14-7. CommonView props 확장 (metaPath / streamPath / downloadPath)

CommonView 모달은 기본적으로 차량 서류 API 경로를 사용하지만,  
아래 선택 props로 API 경로를 오버라이드할 수 있습니다.

```jsx
<CommonView
  metaPath="/api/driver/qual-cert/meta"        // 기본: /api/common-view/bus-document/meta
  streamPath="/api/driver/qual-cert/file"       // 기본: /api/driver/bus-documents/file
  downloadPath="/api/driver/qual-cert/download" // 기본: /api/common-view/bus-document/download
/>
```

이 구조 덕분에 CommonView는 **차량 서류**, **자격증 사본** 등 다양한 파일 유형을 동일한 뷰어 컴포넌트로 처리할 수 있습니다.

## 15. 이번 정리에서의 DB / API 주의 (추가 컬럼 없음)

- **새 DB 컬럼을 임의로 추가하지 않았습니다.** [`BusTaams 테이블.md`](./BusTaams%20테이블.md)에 이미 정의된 **`TB_DRIVER_DOCS`**, `TB_FILE_MASTER.FILE_ID`, `TB_DRIVER_INFO`의 `QUAL_CERT_FILE_ID` / `PROFILE_FILE_ID`만 사용합니다.
- API 응답·쿼리 파라미터의 **`fileId`** 는 모두 **문자형 `FILE_ID` (varchar(20) 0패딩)** 를 가리킵니다. (과거 문서/코드에 남은 `*Uuid` 명칭은 **값이 UUID가 아님** — 혼동 방지로 `*FileId` / `fileId` 로 통일.)
- **`TB_AUCTION_REQ.TRAVELER_ID`**, **`TB_BUS_RESERVATION.DRIVER_ID`**, **`TB_TRIP_REVIEW`의 `WRITER_ID`/`DRIVER_ID`** 등 VARCHAR 정책은 본 화면 API와 별도로, 견적·리뷰·역경매 쿼리에서 이미 `BusTaams 테이블.md`와 맞게 쓰는 것이 목표이며, 기사 프로필 API는 **기사/회원 키를 `userId` 쿼리·JSON 필드**로 전달하는 패턴을 유지합니다.
