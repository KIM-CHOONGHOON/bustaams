# 기사정보등록 (Driver Profile Setup) 화면 명세

본 문서는 `busTaams` 프로젝트의 기사 정보 등록 화면에 대한 UI·데이터·스토리지·외부 연동을 정의합니다. **GCS 버킷·폴더 구조**는 [`ARCHITECTURE.md`](./ARCHITECTURE.md)의 **「파일 및 스토리지 아키텍처」**를 기준으로 합니다.

---

## 1. 화면 개요

- **목적:** 기사 전문성 검증을 위한 면허·자격·프로필 등 추가 정보 수집
- **접근:** 드라이버 대시보드 > 바로가기 > **「기사 정보」**
- **구현:** `busTaams_web/src/components/DriverProfileSetup/DriverProfileSetup.jsx`
- **디자인:** The Radiant Traveler 가이드라인

---

## 2. 업로드 파일 — Google Cloud Storage (GCS)

버킷 기본값: `bustaams-secure-data` (리전 `asia-northeast3`). 업로드는 **백엔드**에서 수행하며, DB에는 `TB_FILE_MASTER`에 메타데이터만 저장합니다.

| 구분 | GCS 객체 prefix (예시) | 비고 |
| :--- | :--- | :--- |
| **프로필 사진** | `profiles/driver/{YYYY-MM}/{FILE_UUID}.png` | `FILE_CATEGORY`: `DRIVER_PROFILE_PHOTO` |
| **운송종사자 자격증 사본** | `certificates/bus_licenses/{YYYY-MM}/{FILE_UUID}.{png\|pdf}` | [`ARCHITECTURE.md`](./ARCHITECTURE.md) **2-2 [기사 서류] 민감 자격 증명서** — `certificates/bus_licenses/` 와 동일 계열. `FILE_CATEGORY`: `BUS_QUAL_CERT` |

- **접근 권한:** Private — 운영 시 Signed URL 또는 백엔드 경유 열람.
- **금지:** DB BLOB에 Base64 원문 적재 (아키텍처 정책과 동일).

---

## 3. 주민등록번호 입력 규칙

- **수집 형식:** **앞 6자리 + 하이픈 + 뒤 1자리**만 허용 (예: `900101-1`).
- **의미:** 생년월일(앞 6자리)과 성별·내·외국인 등을 나타내는 **뒷자리 첫 번째 숫자 1자리**만 입력받아 불필요한 식별자 수집을 줄입니다.
- **저장:** 서버에서 `AES-256-GCM` 양방향 암호화 후 `TB_DRIVER_INFO.RRN_ENC` 등에 저장합니다.
- **검증:** API `POST /api/driver/profile-setup` 에서 정규식 `^\d{6}-\d{1}$` 로 검사합니다.

---

## 4. 휴대전화 인증 — Firebase Phone Authentication

기사 정보 화면과 **회원가입** 화면 모두 **Firebase Authentication(Phone)** 으로 SMS를 보내고, 인증 완료 후 발급되는 **Firebase ID 토큰**을 백엔드(`firebase-admin`)가 검증하는 흐름을 맞춥니다. 서버 쪽 검증 함수는 **`verifyFirebasePhoneIdTokenIfRequired`** 하나로 **`POST /api/auth/register`** 와 **`POST /api/driver/profile-setup`** 에 공통 적용됩니다.

### 4-1. 회원가입 화면 (`SignupPage.jsx`)

| 단계 | 내용 |
| :--- | :--- |
| **초기화** | `App.jsx` 에서 `initializeApp(firebaseConfig)` 후 `window.auth = getAuth(app)` , `window.firebaseAuth = { RecaptchaVerifier, signInWithPhoneNumber }` 를 노출합니다. |
| **reCAPTCHA** | `#recaptcha-signup-wrapper` 컨테이너에 대해 `new RecaptchaVerifier(window.auth, 'recaptcha-signup-wrapper', { size: 'invisible' })` 를 생성합니다. |
| **SMS 발송** | `signInWithPhoneNumber(window.auth, E.164번호, recaptchaVerifier)` — 국내 번호는 `+82` 로 변환(예: `010…` → `+8210…`). |
| **코드 확인** | `confirmationResult.confirm(인증번호)` → `user.getIdToken()` 으로 ID 토큰 확보 → **`signOut(window.auth)`** 로 웹 세션의 Firebase 로그인 상태를 정리(회원가입은 자체 DB 로그인을 쓰므로). |
| **가입 요청** | `POST /api/auth/register` 본문에 **`firebaseIdToken`** 필드로 위 토큰을 실어 보냅니다. |
| **로컬 생략** | `.env`에 `VITE_SKIP_FIREBASE_PHONE=true` 이면 SMS 호출을 건너뜁니다. 이 경우 **서버에 `FIREBASE_SERVICE_ACCOUNT_PATH` 가 없을 때만** 가입이 허용됩니다(Admin이 있으면 토큰 검증 실패). |

> 이전에는 **테스트 모드**로 `signInWithPhoneNumber` 호출이 주석 처리되어 있었고, SMS 없이 `isPhoneVerified` 만 `true` 로 두는 방식이었습니다. 현재는 위와 같이 **실제 SMS 경로를 기본**으로 두고, 필요 시에만 `VITE_SKIP_FIREBASE_PHONE` 으로 생략합니다.

### 4-2. 운전 기사 기본 정보 (`DriverProfileSetup.jsx` + `firebasePhoneVerify.js`)

- **발송:** `signInWithPhoneNumber` 로 SMS 발송. reCAPTCHA는 **invisible** 컨테이너(`#driver-phone-recaptcha`)를 사용합니다.
- **인증 앱:** 로그인용 기본 Firebase 앱과 충돌하지 않도록 `firebasePhoneVerify.js` 에서 **별도 앱 이름**(`phoneVerifyDriver`)으로 `Auth` 인스턴스를 분리합니다.
- **확인:** 6자리 입력 후 **「인증 확인」** → `confirmationResult.confirm(code)` → ID 토큰 → `signOut(phoneAuth)` 로 해당 전용 Auth 세션 정리.
- **서버:** 제출 시 **`phoneIdToken`** 필드로 전송. `POST /api/driver/profile-setup` 에서 회원가입과 **동일한** `verifyFirebasePhoneIdTokenIfRequired` 로 검증합니다.
- **Admin 미설정:** 토큰 없이도 저장 가능(로컬 개발). **Admin 설정 시**에는 토큰 필수.

### 4-3. 백엔드 공통 (`busTaams_server/server.js`)

- **`firebaseAdminConfigured()`** — `FIREBASE_SERVICE_ACCOUNT_PATH` 파일 존재 여부.
- **`verifyFirebasePhoneIdTokenIfRequired(idToken)`** — Admin이 설정된 경우에만 `admin.auth().verifyIdToken(idToken)` 실행; 미설정이면 검증 생략.
- **적용 API:** `POST /api/auth/register` (`firebaseIdToken`), `POST /api/driver/profile-setup` (`phoneIdToken`).

- **참고:** [`ARCHITECTURE.md`](./ARCHITECTURE.md) **외부 서비스 연동 — Firebase Phone Auth** 절.

## 5. 운전면허 정보 — 진위 확인 (백엔드 연동)

**목적:** 면허번호·성명·생년월일·**암호일련번호** 등을 대조하여 유효 면허 여부 확인.

| 항목 | 내용 |
| :--- | :--- |
| **데이터 출처** | 도로교통공단·경찰청 등(서비스별 상이) |
| **공공 오픈 API** | **「운전면허정보 자동검증시스템」** 은 자동차대여사업자 등 **승인된 기관용 B2B** 로 공개되는 경우가 많아, 포털 JSON 한 종류로 고정하기 어렵습니다. |
| **본 프로젝트 구현** | `busTaams_server/driverVerification.js` 의 **`verifyKoroadDriverLicense`** — 기관에서 받은 **전용 검증 URL**(`KOROAD_LICENSE_VERIFY_URL`)로 **JSON POST** 합니다. 응답에 `success`·`valid`·`resultCode === '00'` 등 중 하나가 성공으로 인정되도록 처리합니다(실제 스키마는 연동 기관 명세에 맞게 조정). |
| **활성화** | `KOROAD_LICENSE_VERIFY_ENABLED=true` 일 때만 저장 전에 호출합니다. **기본값은 비활성**이라 기존 배포는 그대로 동작합니다. |
| **입력 필드** | 면허번호(`licenseNo`), 암호일련번호(`licenseSerialNo`), 성명(`driverName`), 주민번호(앞6+뒤1)로부터 유도한 **생년월일 8자리** |

## 6. 버스운전 자격번호 — 진위 확인 (한국교통안전공단 / 공공데이터포털)

**목적:** 버스·택시·화물 등 **운수종사 자격** 유효 여부 확인.

| 항목 | 내용 |
| :--- | :--- |
| **제공** | 한국교통안전공단 — 공공데이터포털 **「한국교통안전공단_운수종사자 자격증 진위여부 확인 API」**(서비스 ID 예: B552016 계열, 상세는 포털 Swagger 참고) |
| **검증 입력** | 성명, 생년월일(YYYYMMDD), 자격증 번호 |
| **본 프로젝트 구현** | `driverVerification.js` 의 **`verifyTsWorkerQualification`** — `GET` 요청으로 `serviceKey` 및 쿼리 파라미터를 붙여 호출합니다. 기본 파라미터명은 `flnm`, `brdt`, `qlfcLcnsNo`(포털 명세와 다르면 `.env`에서 변경). |
| **활성화** | `TS_QUAL_VERIFY_ENABLED=true` 이고 `TS_QUAL_VERIFY_URL`·`PUBLIC_DATA_SERVICE_KEY`(또는 `DATA_GO_KR_SERVICE_KEY`)가 있을 때 저장 전에 호출합니다. **기본값은 비활성**입니다. |
| **생년월일** | 화면에 별도 입력 없이, 주민번호 앞 6자리·**뒷자리 첫 숫자**로 세기(1900년대/2000년대)를 구분해 `YYYYMMDD` 를 계산합니다(`birthYmdFromRrn`). |

| 항목 | 내용 |
| :--- | :--- |
| **조회** | 화면 마운트 시 `GET /api/driver/profile-setup?userUuid=` 호출 |
| **`exists: false`** | `TB_DRIVER_INFO`에 행이 없음 — `TB_USER` 기준 **성명·휴대전화**만 채움. 면허·자격 필드는 신규 입력(잠금 없음) |
| **`exists: true`** | 주민번호 앞6·뒤1, 면허(종류·번호·암호일련·발급·만료), 버스운전 자격번호, 자기소개 등으로 폼을 채움. **운전면허 정보**·**운송종사자 자격**(자격번호·자격증 업로드 영역)은 기본 **읽기 전용**이며, 각 섹션의 **「수정」**으로 잠금 해제 후 편집 가능 |
| **프로필·자격 파일** | `hasProfilePhoto` / `hasQualCertFile`이 true이면, 새 파일을 선택하지 않아도 기존 서버 파일이 유지됨(안내 문구 표시) |

### 7-4. 진위 검증 생략 규칙 (백엔드)

저장(`POST`) 직전, 서버는 `TB_DRIVER_INFO`에서 조회한 **기존 행**과 이번 요청 본문을 비교합니다(`busTaams_server/driverVerification.js`).

| 구분 | 비교 필드 | 동일할 때 |
| :--- | :--- | :--- |
| **운전면허 진위 (KOROAD)** | `LICENSE_TYPE`, `LICENSE_NO`, `LICENSE_SERIAL_NO`, `LICENSE_ISSUE_DT`, `LICENSE_EXPIRY_DT` | `KOROAD_LICENSE_VERIFY_ENABLED=true`여도 **면허 진위 API를 호출하지 않음** |
| **운수종사자 자격 (TS)** | `QUAL_CERT_NO`(버스운전 자격번호) | `TS_QUAL_VERIFY_ENABLED=true`여도 **자격 진위 API를 호출하지 않음** |

- **최초 등록:** 기존 행이 없으면 위 비교가 성립하지 않으므로, 플래그가 켜져 있으면 진위 검증을 수행합니다.
- **수정:** 화면에서 「수정」을 눌렀다가 **값을 바꾸지 않고** 제출해도, 요청 값이 DB와 같으면 동일하게 **진위 API를 생략**합니다. 면허 또는 자격번호 중 **하나라도** 바뀌면 해당 항목만 외부 검증 대상이 됩니다.

### 7-5. 하단 제출 버튼 · 성공 모달

| 항목 | 내용 |
| :--- | :--- |
| **버튼 문구** | `GET` 결과 `exists: false`(최초 등록)일 때 **「등록」**, `exists: true`(이미 `TB_DRIVER_INFO`에 행 있음)일 때 **「수정」**. 제출 중에는 **「처리 중...」** |
| **성공 후 동작** | 저장 성공 시 **대시보드 등으로 자동 이동하지 않음**. 같은 화면에 머무르며 **모달(팝업)** 으로만 안내합니다. |
| **최초 등록 성공** | 모달 메시지: **「버스 기사 기본 정보가 등록되었습니다.」** — 확인 시 모달만 닫힘. 이후 동일 세션에서도 `profileExistsOnServer` 상태를 갱신하여 주요 버튼이 **「수정」**으로 바뀌고, 면허·자격 블록은 다시 잠금 처리됩니다. |
| **수정 성공** | 모달 메시지: **「버스 기사 기본 정보가 수정 되었습니다.」** |
| **취소** | **「취소」**는 기존과 같이 상위에서 넘긴 `onBack`으로 대시보드 등 이전 화면으로 돌아갑니다(제출 버튼과 무관). |

## 8. 백엔드 API — `GET /api/driver/profile-setup`

- **목적:** 등록된 기사 정보를 불러와 폼에 채움.
- **쿼리:** `userUuid` (필수)
- **동작:** `TB_USER`에서 성명·휴대전화(복호화), `TB_DRIVER_INFO`에서 면허·자격·자기소개·파일 UUID 존재 여부. 테이블/행이 없으면 `exists: false`와 함께 사용자 정보만 반환할 수 있음.

**응답 필드 (요약):**

| 필드 | 설명 |
| :--- | :--- |
| `exists` | `TB_DRIVER_INFO`에 해당 사용자 행이 있으면 `true` |
| `userName`, `phoneNo` | `TB_USER` 기준(복호화) |
| `rrnFront`, `rrnBack` | 주민번호 앞 6자리 / 뒤 1자리 (`RRN_ENC` 복호화 후 파싱) |
| `licenseType`, `licenseNo`, `licenseSerialNo`, `licenseIssueDt`, `licenseExpiryDt` | 면허 정보 |
| `qualCertNo` | 버스운전 자격번호 |
| `bioText` | 자기소개 |
| `hasProfilePhoto`, `hasQualCertFile` | 각각 프로필 사진·자격증 파일 UUID 존재 여부 |

## 9. 백엔드 API — `POST /api/driver/profile-setup`

- **기능:** 주민번호 형식 검증, **Firebase ID 토큰 검증**(`verifyFirebasePhoneIdTokenIfRequired`, 회원가입과 동일), **선택적 면허·운수자격 진위**(`runDriverVerificationsForProfileSetup` → `driverVerification.js`), GCS 업로드, `TB_FILE_MASTER`·`TB_DRIVER_INFO` Upsert  
- **테이블 보장:** `TB_DRIVER_INFO` / `TB_FILE_MASTER` 가 없으면 서버가 `CREATE TABLE IF NOT EXISTS` 로 생성(아래 **11-2**).  
- **`LICENSE_SERIAL_NO`:** 예전 DB에만 해당 시 `ALTER TABLE` 로 컬럼 추가(이미 있으면 무시).

**회원가입 API와의 정합:** `POST /api/auth/register` 역시 `firebaseIdToken` 에 대해 동일 함수로 검증하며, `TB_USER.SMS_AUTH_YN` 은 인증 통과 시 `Y` 로 저장합니다.

**Request Body (예):**

```json
{
  "userUuid": "회원 UUID 문자열",
  "driverName": "홍길동",
  "phoneIdToken": "Firebase ID 토큰(서비스 계정 설정 시 필수)",
  "rrn": "900101-1",
  "licenseType": "1종 대형",
  "licenseNo": "12-34-567890-11",
  "licenseSerialNo": "암호일련번호",
  "licenseIssueDt": "2023-01-01",
  "licenseExpiryDt": "2033-01-01",
  "qualCertNo": "자격번호",
  "bioText": "...",
  "profilePhotoBase64": "data:image/png;base64,...",
```

## 10. 면허·자격 진위 API 환경 변수 (`busTaams_server/.env`)

`busTaams_server/driverVerification.js` 가 읽습니다. **둘 다 끄면**(기본) DB 저장만 하고 외부 호출은 하지 않습니다.

### 10-1. 한국교통안전공단 — 운수종사자 자격증 진위 (공공데이터포털)

| 변수 | 설명 |
| :--- | :--- |
| `TS_QUAL_VERIFY_ENABLED` | `true` 일 때만 자격 진위 API 호출 |
| `PUBLIC_DATA_SERVICE_KEY` 또는 `DATA_GO_KR_SERVICE_KEY` | 공공데이터포털에서 발급한 **일반 인증키**(디코딩된 값) |
| `TS_QUAL_VERIFY_URL` | 포털 **상세 기능 명세**에 나온 **요청 URL 전체**(예: `https://apis.data.go.kr/B552016/.../xxx`) |
| `TS_QUAL_PARAM_NAME` | (선택) 성명 쿼리 키, 기본 `flnm` |
| `TS_QUAL_PARAM_BIRTH` | (선택) 생년월일 쿼리 키, 기본 `brdt` |
| `TS_QUAL_PARAM_CERT` | (선택) 자격증번호 쿼리 키, 기본 `qlfcLcnsNo` |

호출 시 `_type=json` 을 기본으로 붙입니다. 응답은 JSON 또는 XML 모두 처리하며, `resultCode` 가 `00` 인지 확인합니다.

### 10-2. 운전면허 진위 (기관 전용 URL / 게이트웨이)

도로교통공단 **자동검증 OpenAPI** 는 기관 승인·엔드포인트가 제각각일 수 있어, **고정 URL 대신** 아래처럼 **연동 받은 주소**를 넣는 방식입니다.

| 변수 | 설명 |
| :--- | :--- |
| `KOROAD_LICENSE_VERIFY_ENABLED` | `true` 일 때만 면허 진위 호출 |
| `KOROAD_LICENSE_VERIFY_URL` | JSON을 받는 **검증 API URL**(POST 기본) |
| `KOROAD_LICENSE_HTTP_METHOD` | (선택) 기본 `POST` |
| `KOROAD_LICENSE_API_KEY` | (선택) `Authorization: Bearer …` 로 붙일 토큰 |

요청 본문(JSON): `{ "name", "birthYmd", "licenseNo", "licenseSerialNo" }` — 실제 기관 스펙과 다르면 `driverVerification.js` 내 페이로드/성공 판정 로직을 수정해야 합니다.

## 11. 데이터베이스

### 11-1. 논리 스키마 (문서)

- 상세 정의는 `BusTaams_Project 테이블 설계.md` **4. 기사 전문 정보 관리 (`TB_DRIVER_INFO`)** 및 **1. `TB_FILE_MASTER`** 를 따릅니다.
- **면허 암호일련번호**는 `LICENSE_SERIAL_NO` 컬럼으로 저장합니다.

### 11-2. 물리 테이블 자동 생성 (서버)

로컬·신규 DB처럼 테이블이 아직 없어도 **`POST /api/driver/profile-setup` 호출 시** `busTaams_server/server.js` 가 다음을 수행합니다.

| 순서 | 대상 | 동작 |
| :--- | :--- | :--- |
| 1 | `TB_DRIVER_INFO` | **`CREATE TABLE IF NOT EXISTS`** — PK `USER_UUID`, 면허·자격·프로필 UUID·`BIO_TEXT`·타임스탬프 등 설계서와 동일한 컬럼 세트로 생성 |
| 2 | `TB_FILE_MASTER` | **`CREATE TABLE IF NOT EXISTS`** — 프로필 사진·자격증 사본 업로드 시 GCS 메타데이터 `INSERT` 가 가능하도록 최소 스키마 생성 (`FILE_CATEGORY`는 `VARCHAR(50)` 로 유연하게 수용) |
| 3 | `TB_DRIVER_INFO.LICENSE_SERIAL_NO` | 이미 예전 버전의 테이블만 있을 때 **`ALTER TABLE … ADD COLUMN`** (컬럼이 있으면 무시) |

- 위 DDL은 **트랜잭션(파일 업로드·Upsert) 시작 전**에 실행됩니다(MySQL DDL 특성상 트랜잭션 밖에서 처리).
- 운영 DB에서는 사전에 DBA가 동일 스키마로 마이그레이션해 두는 것이 권장되며, 서버의 `CREATE IF NOT EXISTS` 는 개발·초기 배포 편의용입니다.
